# AXEON サイト プロジェクト方針

## ⚠️ 最重要ルール: 安全第一

このプロジェクトでは、**ユーザー(以下「私」)はGit操作・コマンドラインに不慣れです**。
そのため、以下のルールを**絶対に**守ってください。

### 🔴 絶対禁止事項(明示的な許可がない限り、絶対に実行しない)

以下の操作は、**私が明確に指示し、内容を理解した上でOKを出した場合のみ**実行可能です。
推測や判断で勝手に実行することを禁じます。

- `git reset --hard` (未コミットの変更が消える)
- `git push --force` / `git push -f` (リモート履歴を破壊する)
- `git rebase` 系の操作(履歴を書き換える)
- `git clean` (未追跡ファイルを削除する)
- `git checkout -- <file>` (ファイルの変更を破棄する)
- `rm -rf` などの広範囲のファイル削除
- `mainブランチ` への直接コミットまたは push
- 既存のgitタグの削除や移動
- `.git` ディレクトリへの操作
- 環境変数・認証情報ファイル(`.env*`, `*.pem`, `id_rsa` など)の変更
- `package.json` の依存関係を「削除」する変更(追加はOK、削除は要確認)
- node_modules や dist などの大きなディレクトリの削除

### 🟡 必ず事前に説明と承認が必要な操作

以下の操作は、**実行する前に必ず日本語で平易に説明し、私の承認を得てから**進めてください。

1. **複数ファイルにまたがる変更**(3ファイル以上を同時に変更する場合)
2. **新しいnpmパッケージのインストール**(なぜ必要かと、リスクを説明)
3. **設定ファイルの変更**(`next.config.js`, `tsconfig.json`, `tailwind.config.js` など)
4. **ディレクトリ構造の変更**(フォルダの新規作成・移動・削除)
5. **既存コンポーネントの大幅なリファクタリング**(削除・分割・統合)
6. **データベース・API・外部サービスへの接続変更**

### 🟢 自由に進めて良いこと

以下は説明だけして進めてOKです(承認待ちは不要):

- 単一ファイル内のテキスト・スタイル修正
- 新規ファイルの作成(既存ファイルを壊さない場合)
- コメントの追加・修正
- 軽微なバグ修正(ロジックを大きく変えない範囲)
- READMEなどドキュメントの更新

## 🛡️ 作業開始時の必須プロセス

毎回の作業セッションを以下の手順で開始してください:

### 1. 現状確認

セッションの最初に、以下を必ず実行・報告すること:
```
- 現在のブランチ名
- 直近のコミットハッシュとメッセージ
- 未コミットの変更があるか(あれば内容)
- 未追跡ファイルの有無
```

私が「いつでもこのコミットに戻れる」と確信できる状態を作ってください。

### 2. 作業前の確認

未コミットの変更がある場合:
- 「先にコミットしますか?それとも変更を残したまま作業しますか?」と必ず私に聞いてください
- 私の確認なしで `git stash` も `git add` も実行しないでください

### 3. PLAN提示

実装前に必ず以下を含むPLANを提示してください:

```
## 変更計画

### やること
- [箇条書きで簡潔に]

### 触るファイル
- ファイル1: 変更内容の概要
- ファイル2: 変更内容の概要

### 触らないファイル
- 念のため、影響範囲外のファイルを明示

### 想定リスク
- ビルドが壊れる可能性: 低/中/高
- 既存機能への影響: 具体的な内容

### ロールバック方法
- 失敗した場合の戻し方を1行で

この計画で進めてよろしいですか?
```

私が「OK」「進めて」「Yes」など明確な承認を出した場合のみ、実装に入ってください。

### 4. 実装中

- ファイルを変更する前に、変更内容のdiffを見せてください
- 大量に変更する場合は、**1セクションごとに区切って**実行してください
- エラーが出たら、勝手にリトライせず、**まず私に状況を報告**してください

### 5. 作業終了時

- 変更内容を要約してください
- **コミットメッセージの案**を提示してください(私が承認してから `git commit` を実行)
- pushはしないでください(pushは私が手動で行います)

## 🤔 判断に迷った時のルール

- 判断に迷う点があれば、**勝手に進めず必ず質問してください**
- 私が初心者であることを前提に、**専門用語は避けるか、必ず説明を添える**こと
- 質問する時は、**選択肢を2〜3個**提示してください(「A: 〇〇する / B: △△する / C: 何もしない」のように)
- 「〜だと思いますが、進めて良いですか?」のように、**判断の根拠**も添えてください

## 📚 プロジェクト概要

### サイトの目的
中堅企業以上のDX推進担当者・経営層からの問い合わせ獲得

### ターゲット
- メイン: 従業員200〜3000名規模の中堅企業のDX推進担当者・経営層
- サブ: 大企業の事業部・新規事業責任者
- 補足: 小規模事業者の相談も受け付ける(ただしメインメッセージでは触れない)

### 訴求の核
- 経営課題の整理とロードマップ設計に強いコンサルティング経験者 × AI開発のトップエンジニア（表記ルールは `docs/strategy.md`）
- 戦略から実装まで一気通貫
- AI駆動開発による高速実装

### サイトのトーン
- プロフェッショナル、BtoB信頼感重視
- ダーク+宇宙感の世界観(ブランド資産として維持)
- 派手なアニメーションは避ける
- 経営層が読むため、本文フォントサイズは小さくしない(15px以下にしない)

### ファーストビュー(現在の方針)
- 主見出し: 「戦略実装、一気通貫で。」
- サブ見出し: 「経営課題の整理とロードマップ設計に強いコンサルティング経験者 × AI開発のトップエンジニア。」（長い場合は `docs/strategy.md` の短縮表記を使う）
- リード文: DX戦略の構想から、AI駆動開発による高速実装、内製化の支援まで。
- メインCTA: 「初回コンサルティングのご相談」
- サブCTA: 「サービス詳細」

## 🔧 技術方針

### 既存資産の尊重
- 既存のコンポーネントを最大限再利用してください
- 既存のデザインシステム(色、余白、タイポグラフィ)に合わせてください
- 勝手に新しいライブラリを導入しないでください(必要な時は必ず確認)

### コーディング規約
- 既存ファイルのコードスタイルに従ってください
- 1コミット = 1機能/1セクション の粒度を守ってください
- コミットメッセージは日本語OK、`feat:` `fix:` `style:` などのプレフィックス推奨

## 🆘 こんな時どうする

### 私が「やって」とだけ言ってきた場合
→ 何をやるか曖昧な場合は、**勝手に推測せず**「具体的には何をしますか?」と聞き返してください。

### 私の指示が技術的に不可能/問題がある場合
→ 黙って従わず、**「これこれの理由で問題があります、代わりに〇〇はどうですか?」**と提案してください。

### 何かを壊してしまった場合
→ 隠さず即座に報告してください。`git reflog` で復旧可能なことが多いので、慌てず私に状況を伝えてください。

### 長時間の作業で疲れた/迷子になった場合
→ 「いったん区切りませんか?ここまでの変更をコミットしましょう」と提案してください。
