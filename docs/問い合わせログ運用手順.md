# 問い合わせログ運用手順（公開前〜実メール移行まで）

対象: `api/contact` を「ログ運用」で公開する期間  
参照実装: `src/app/api/contact/route.ts`

## 1. 運用目的

- 問い合わせ取りこぼしを防ぐ
- 公開初期の運用負荷を定量化する
- 実メール送信移行までの暫定運用を標準化する

## 2. 役割

- [ ] 運用責任者（最終判断）
- [ ] 一次確認担当（ログ確認・受付）
- [ ] 技術担当（障害時調査）
- [ ] 代理担当（不在時バックアップ）

## 3. 確認頻度（最低ライン）

- 平日: 1時間ごと
- 公開当日: 15分ごと（公開から24時間）
- 土日祝: 2時間ごと

## 4. 日次運用フロー

1. ログから `"[Contact] inbound"` を抽出
2. 同一時刻帯の `admin mail draft` / `customer mail draft` 出力を確認
3. 受付台帳へ転記（時刻、連絡先、要点、担当者、状態）
4. 担当者アサインと一次返信（手動）
5. 対応完了後、台帳ステータスを更新

## 5. 見逃し防止フロー

- [ ] 1次確認時に未転記件数 0 を確認
- [ ] 未転記があれば 10 分以内に台帳へ反映
- [ ] 未処理が 2 時間以上残る場合は責任者へエスカレーション
- [ ] 1日終了時に「未処理 0 件」を確認

## 6. 送信失敗・障害時フロー

### 失敗判定

- 問い合わせ送信UIで失敗表示が継続
- `api/contact` で 5xx/429 が継続
- ログに inbound が記録されない

### 初動

1. 影響範囲を確認（全体 / 一部）
2. 一時告知（必要時）
3. 代替手段で受付（手動連絡先の案内）
4. 技術担当へ調査依頼
5. 復旧後に受付漏れ有無を再点検

## 7. 取り扱いルール（個人情報）

- [ ] `name` / `email` / 本文相当の共有範囲を限定
- [ ] 台帳のアクセス権を最小化
- [ ] 共有時は必要最小限のみ記載
- [ ] 保存期間と削除手順を明文化

- `api/contact` が実メール送信へ移行し、監視が安定運用化された時点
