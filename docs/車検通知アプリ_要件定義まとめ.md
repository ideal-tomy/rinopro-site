# 車検通知アプリ 要件定義まとめ（草案）

## 1. 背景と目的

- 顧客情報・車両情報をアプリ上で一元管理する
- 車検や整備の適切なタイミングで自動通知を行う
- 通知と同時に見積を提示し、来店・受注につなげる
- 将来的にAIチャットボットで顧客情報変更や問い合わせ対応を可能にする

## 2. 想定する業務フロー

1. 顧客情報・車両情報を登録（手動入力/自動入力）
2. 走行距離や経過日数、車検期限を継続的に管理
3. 条件に応じて通知を自動送信（アプリ通知、LINE通知）
4. 通知時に見積を提示（法定費用と整備費用を分離）
5. 顧客反応に応じて予約・問い合わせへ接続

## 3. 管理データ（DB）

### 3.1 顧客マスタ

- 顧客ID
- 氏名
- 電話番号/メールアドレス
- LINE連携ID
- 通知許可設定（チャネル別）

### 3.2 車両マスタ

- 車両ID
- 顧客ID（紐付け）
- 車体番号
- 車両登録番号
- 初年度登録
- 型式
- 現在走行距離
- 次回車検満了日

### 3.3 整備・車検履歴

- 履歴ID
- 車両ID
- 実施日
- 実施内容
- 使用部品
- 費用
- 次回推奨時期（距離/日付）

### 3.4 見積データ

- 見積ID
- 車両ID
- 作成日時
- 法定費用
- 整備費用
- 合計
- 計算根拠（適用した社内ナレッジ）

## 4. 入力要件

- 手入力で基本情報を登録できる
- 車検証情報などの自動入力（OCR/API連携）を将来拡張できる
- 必須項目チェック、重複チェック、形式チェックを行う
- 走行距離は更新履歴を保持できる

## 5. 通知要件（Webhook）

### 5.1 通知チャネル

- アプリ内通知
- LINE通知
- 将来拡張: メール、SMS

### 5.2 通知トリガー

- 車検満了日の一定日前（例: 90日/30日/7日前）
- 走行距離の閾値到達（例: 5,000kmごと）
- 前回整備からの経過期間（例: 6か月/12か月）

### 5.3 通知制御

- Webhook経由で通知基盤へ送信
- 送達結果（成功/失敗）をログ保存
- 失敗時の再送制御
- 通知テンプレート管理（チャネル別）

## 6. 見積自動計算要件

- 社内ナレッジ（工賃、部品相場、車種別パターン）を参照して自動計算する
- 法定費用と整備費用を分けて表示する
- 計算の内訳・根拠を保存し、説明可能にする
- 通知文面に見積サマリを差し込めるようにする

## 7. AIチャットボット要件（将来）

- 顧客側からの情報変更依頼をチャットで受け付ける
- 変更処理には本人確認・権限確認を行う
- 社内ナレッジを参照して回答する
- 対応不能な内容は有人対応へエスカレーションする
- 変更履歴・監査ログを保持する

## 8. 非機能要件（最低限）

- セキュリティ: 個人情報保護、アクセス制御、通信暗号化
- 可用性: 通知失敗時リトライ、障害時復旧手順
- 保守性: 通知条件・見積ルールを運用で更新可能
- 拡張性: OCR、外部システム連携、通知チャネル追加に対応
- 監査性: 変更ログ、通知ログ、見積生成ログを追跡可能

## 9. MVP（初期リリース範囲）

- 顧客/車両情報の登録・編集
- 車検期限ベースの自動通知（アプリ内 + LINE）
- 見積自動生成（法定費用 + 基本整備費用）
- 通知履歴と配信結果を管理画面で確認

## 10. 今後の詳細化項目（要件定義で詰める項目）

- 通知タイミングの具体ルール（業務運用に合わせる）
- 見積ロジックの優先順位と例外条件
- LINE通知の運用設計（同意取得、停止導線）
- AIチャットの適用範囲（閲覧のみ/更新まで）
- 既存システムとのデータ連携方式
