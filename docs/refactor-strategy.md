# AXEON サイト整合化リファクタ計画

作成日: 2026-05-06
前提資料: `docs/strategy.md`, `docs/redesign-2026.md`, `docs/site-audit-2026.md`

---

## 0. 全体方針

### 哲学: 「最小限の整合化、構造は触らない」

旧「AIコンシェルジュ主導型」コンセプトの残骸を、**機能としては残しつつ、UIと文言から戦略と矛盾する部分を排除する**。

- ❌ やらない: 大規模な構造改革、コンポーネント削除、機能廃止
- ✅ やる: 文言の整合化、UIの位置づけ調整、目立たないようにする

### ターゲット状態(リファクタ完了時)

- サイト全体が `strategy.md` と整合した状態
- AIコンシェルジュ機能は動作するが、UIから「主役感」が消えている
- トップページの違和感3つが解消されている
- 旧コンセプトの未使用ファイルに「廃止候補」コメントが付いている
- 営業活動を開始できる完成度

## 1. Phase 構成

### Phase A: 文言の整合化(低リスク・最優先)
**所要時間目安**: 1〜2時間  
**リスク**: 低(文字列置換中心)  
**目的**: 戦略と矛盾する文言を整える

### Phase B: UI位置づけ調整(中リスク)
**所要時間目安**: 1〜2時間  
**リスク**: 中(スタイル変更)  
**目的**: コンシェルジュ系UIを目立たなくする

### Phase C: トップページ違和感修正(中リスク)
**所要時間目安**: 2〜3時間  
**リスク**: 中(コンポーネント調整)  
**目的**: 既知のUI違和感3つを解消

### Phase D: 旧コンセプト残骸への目印(極低リスク)
**所要時間目安**: 30分  
**リスク**: 極低(コメント追加のみ)  
**目的**: 将来の整理対象を明示

## 4. Phase C: トップページ違和感修正

### C-1. セクションナビの位置・固定方式の整理

**問題**: 
- ファーストビュー直下にナビが浮いている
- スクロール時の挙動が中途半端

**対象ファイル**: 要調査(Claude Codeで特定)
- 推測候補: `src/components/home/HomeStickyNav.tsx` または同等

**変更内容**:

#### C-1-1. 固定方式
```
sticky 実装で、画面上部に常時表示:
- top: 0(ヘッダーがある場合は ヘッダーの高さ)
- z-index: ヘッダーより下、コンテンツより上
- backdrop-filter: blur(12px) で背景を透過
- background: rgba(0,0,0,0.6) 程度
```

#### C-1-2. アクティブ状態
- 現在表示中のセクションをハイライト
- アクセント色のアンダーライン or 背景色

#### C-1-3. モバイル対応
- 小さい画面ではボタンを横スクロールで配置
- タップ可能な高さを確保(44px以上)

### C-2. サービスの流れカードの再設計

**問題**: 4ステップカードが縦長すぎる

**対象ファイル**: 要調査(推測: `src/components/home/HomeProcessCards.tsx` または同等)

**変更内容**:

#### C-2-1. カード内容の整理
```
現状: 各カード内に「コンサル視点」「AI・開発視点」が縦に積まれて長い

改善案A: 視点を横並びにする(2カラム)
改善案B: 1つの視点に統合し、本文を簡潔に
改善案C: 本文を短くしてカードを正方形に近づける
```

**推奨**: 改善案C(本文短縮 + カードを正方形に近く)

#### C-2-2. ステップ間の流れの可視化
- 矢印「>」を大きく、目立つ色で
- もしくは番号「1 → 2 → 3 → 4」を強調

### C-3. 「業種別・業務別の入口」階層整理

**問題**: 親見出しと子見出しが意味的に重複

**対象ファイル**: 要調査(推測: `src/components/home/HomeSeoEntrySection.tsx` または同等)

**変更内容**:

#### C-3-1. 階層の整理(2案から選択)

**案A: 統合(タブ形式)** ← 推奨
```
[セクション見出し] 業種・業務から探す
[リード文] 自社に近い文脈から、課題と打ち手のイメージを掴めます
[タブ] [業種から] [業務から]
[コンテンツ] (選択タブの内容)
```

**案B: 完全分離(2セクション)**
```
[セクション 05] 業種から探す
[セクション 06] 業務から探す
```

**推奨**: 案A(タブ形式)
理由: スクロール量削減、情報の比較がしやすい

#### C-3-2. 余白の整理
- 大きすぎる空白を削減
- 子見出しを廃止する場合、リード文との余白を最適化

## 6. 実装ルール

### 6-1. 各Phase開始時のチェック

各Phaseを開始する前に必ず:
1. 現在のブランチを確認(mainではないこと)
2. 未コミットの変更がないか確認
3. 必要なら新しい作業ブランチを切る:
   - Phase A: `refactor/phase-a-copy-alignment`
   - Phase B: `refactor/phase-b-ui-deemphasize`
   - Phase C: `refactor/phase-c-top-fixes`
   - Phase D: `refactor/phase-d-deprecation-marks`

### 6-2. 各タスクの作業手順

1. **PLAN提示**: 該当する変更箇所を明確化(このドキュメント参照)
2. **承認待ち**: ユーザーの確認を待つ
3. **実装**: 1タスク = 1〜数ファイルの変更
4. **diff提示**: 変更内容を見せる
5. **コミット提案**: コミットメッセージ案を提示
6. **承認後コミット**: ユーザーOK後にコミット
7. **次のタスクへ**

### 6-3. コミットメッセージ規約

```
Phase A: refactor(copy): 〜
Phase B: refactor(ui): 〜
Phase C: refactor(home): 〜
Phase D: docs: 〜
```

例:
- `refactor(copy): フッター文言を戦略整合に更新`
- `refactor(ui): AIコンシェルジュFABを控えめサイズに変更`
- `refactor(home): セクションナビをsticky topに固定`

### 6-4. やってはいけないこと

- ✗ chat/ ディレクトリのファイル削除
- ✗ コンシェルジュ機能の動作変更(UIスタイル以外)
- ✗ コンポーネントの大規模リファクタ
- ✗ ファイル削除全般
- ✗ mainブランチへの直接コミット
- ✗ 複数Phaseの同時進行

### 6-5. 判断に迷った時

- 文言の言い換えで迷った場合 → strategy.md の方針に立ち返る
- UI調整の度合いで迷った場合 → 「目立たなくする」「機能は残す」を基準に
- 影響範囲で迷った場合 → ユーザーに質問(勝手に範囲を広げない)

## 7. 進め方

### 推奨順序

**Day 1**: Phase A(文言整合)
- 一番効果が見える、リスクも低い
- これだけでサイトが「まとも」になる

**Day 2**: Phase D(deprecation コメント)
- 短時間で終わる
- コードベースの方針が明確になる

**Day 3**: Phase B(UI位置づけ調整)
- スタイル変更、見た目の最終調整

**Day 4**: Phase C(トップ違和感修正)
- 一番大きな変更
- 構造に手を入れるのでリスク中

### 並行作業の禁止

- 1つのPhaseを完了させてから次のPhaseに進む
- 1日に2つのPhaseをまたいで作業しない

### リリース判断のタイミング

- Phase A完了 → 戦略整合性確保(リリース可能レベル)
- Phase B完了 → UIの違和感ほぼ解消(推奨リリースタイミング)
- Phase C完了 → ベストな状態
- Phase D → コードベース整理(リリース判断には影響しない)

## 8. 影響範囲とロールバック

### 影響範囲

| Phase | 影響範囲 | 機能への影響 |
|-------|---------|------------|
| A | 全ページの文言 | なし(文字列のみ) |
| B | コンシェルジュ系UI | UIのみ、機能維持 |
| C | トップページ | トップページのみ |
| D | 未使用ファイル | なし(コメントのみ) |

### ロールバック方法

各PhaseはGitブランチで分離されているため:
- 個別Phase取り消し: `git checkout main && git branch -D refactor/phase-X-...`
- 個別コミット取り消し: `git revert <commit-hash>`
- 緊急ロールバック: 該当ブランチを破棄、mainから再開

## 11. Phase E: ナビゲーションフィードバック改善(全ページ共通)

### 背景
クリックからページ遷移までの間、ユーザーへのフィードバックが弱く、
「押せているか分からない」「壊れているのか?」と感じる状態。
これは離脱率に直結する致命的な UX 問題。

### 目的
全ページのリンク・ボタンクリック時に、明確なフィードバックを提供し、
キビキビした操作感を実現する。

### E-1. グローバルローディングバーの導入

**実装内容**:
- ページ遷移時に画面最上部に細い線のプログレスバーを表示
- アクセント色(現状の青)で表示
- 高さ 2〜3px、画面幅いっぱい

**実装方法**:
推奨: `next-top-loader` パッケージを導入
```bash
npm install nextjs-toploader
```

または自前実装: Router events を監視して状態管理

**配置**: app/layout.tsx で全ページに適用
