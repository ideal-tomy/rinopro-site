# 詳細見積ウィザード：コア軸必須化の画面フロー案

[0401見積もりUIUX見直し.md](./0401見積もりUIUX見直し.md)・[`estimate-detailed-narrow-eligibility.ts`](../src/lib/estimate/estimate-detailed-narrow-eligibility.ts)・[`estimate-wizard-step-visibility.ts`](../src/lib/estimate/estimate-wizard-step-visibility.ts) と整合させるための **設計メモ**（実装タスク分解用）。

---

## 1. 目的

- **見積レンジを選択式の組み合わせで説明可能にする**ため、レンジ計算に使う軸では **「未選択のまま進めない」** 状態に寄せる。
- **レイアウトや見た目の UIUX は変えず**、入力コントロールの種類（選択必須・任意補足）と **サーバー側ロジック**を変える。
- **定性情報（短い補足・うまくいっていないことの選択 等）は、計算に使わない。** 要件定義・親切な説明文・フォローアップ文の生成にだけ使う（現状の「文章を出す」体験は維持しつつ、数値はルールで決める前提）。

### 1.1 コピー方針（採用／非採用）

**採用しない:** 各ステップのヒントを **「見積の幅を現実に近づけるため」** と **「あとで説明文に使うだけ」** の二役に分けて短く統一する書き方。必須質問が増える前提では、役割の二分説明はかえって負担に感じられやすいため **使わない**。

**採用する:** **精度の説明**に寄せる。すなわち「この回答で **何が確定し、見積のどの部分がより現実に近づくか**」を短く書く。説明の軸の例として **規模（開発の型）・人数・連携・業種** を中心に示し、**現場の細かい条件**によっても変わり得ることは一言触れる。

**本見積への導線:** このウィザードでは **本見積の前提までここでは詰め切らない** ことを明示する。**問い合わせ**（および想定する **本見積用の詳細フォーム／すり合わせ**）で深掘りする前提とし、ここでは **説明・たたき台・目安のレンジ** にとどまる旨をコピーで示す。必須項目が多い分、**「全部ここで決めなくてよい」** が伝わるようにする。

## 2. 成立させたいパイプライン（ゴール像）

```text
質問の回答状態（選択中心・コア必須）
  → 要件定義本文の材料（テンプレ差し込み ± 生成で整形）
  → 見積レンジ（ルールテーブル＋係数。補足テキストは参照しない）
  → 補足説明・概略・フォローアップ等（現状と同様の生成／表示。数値は上で確定した値に合わせるのみ）
```

**本ドキュメントの範囲**は主に **左端（質問状態・Tier）** と、**§12（採用予定・数値の扱い・実装チェック）**。

## 3. コア軸の定義（Tier）

### Tier A — ルールベース見積・狭帯判定と直結（必須候補）

`isNarrowRangeEligible` が参照する日本語キーと、人数・時期・業種に相当する項目。

| questionId | 画面ラベル（現行） | 備考 |
|------------|-------------------|------|
| `industry` | 業種 | handoff でスキップ時は bundle 済みで OK |
| `summary` | いまいちばんやりたいこと・課題 | **開発の型（単一選択）必須 ＋ 任意の短い補足**へ変更（§3.1）。補足は **計算に使わない**。 |
| `teamSize` | 会社やチームの人数のイメージ | `profileFromEstimateAnswers`（大規模判定） |
| `timeline` | いつ頃までに、という希望 | レンジの時間圧縮イメージ用 |
| `integration` | 今お使いのツールや、他のシステムとのつなぎ | 非表示ステップの前提に効く |
| `hostingContext` | データやシステムの置き場所のイメージ | **`integration` の直後**。UI に「使用環境」という語は使わない。狭帯判定のキーに含める（実装済み）。ルール見積の係数は §12.2 のたたき台で後追い可。 |
| `usageSurface` | 主な使い方・載せる場所 | `hostingContext` と役割分担に注意（到達範囲 vs チャネル） |
| `dataSensitivity` | 扱う情報に個人情報は含まれますか | 狭帯ブロック条件にも関与 |
| `audienceScope` | 誰が使う・見るか（社内・外部） | 同上・login 非表示条件 |
| `currentWorkflow` | いまの情報の扱い方（中心） | |
| `updateFrequency` | 情報の更新の頻度 | **単体運用時はステップ非表示**（§5） |
| `designExpectation` | 見た目・デザインの期待 | |
| `loginModel` | ログインの使い方 | **条件でステップ非表示**（§5） |

### 3.1 `summary` の変更方針（確定イメージ）

- **必須**: 開発アーキタイプを **1つ選択**（例: 業務管理ダッシュボード／自動化ツール／ナレッジ共有ツール … ※型一覧は **§12.2**）。
- **任意**: **短い補足**（自由記述）。**見積の上下限・係数には入れない。** 要件定義・説明文生成の入力にだけ使う。
- フォーム上は既存の「やりたいこと」ステップの **コンポーネントを差し替えるだけ**（同じステップ位置・同じフロー感）。

### Tier B — 比較・説明用（計算から分離）

| questionId | 備考 |
|------------|------|
| `budgetBand` / `budgetFeel` | 価格推定プロンプトからは既に分離済みの設計。必須化はプロダクト判断。 |
| `pain` | **選択式に変更**（程度・パターンのラジオ等）。**計算に使わない**。**任意**（§12.2）。 |
| `constraints` | 当面は任意の自由記述のままでも可。将来選択式化は別タスク。 |

## 4. 画面フロー（ステップ順・現行定義どおり）

[`ESTIMATE_WIZARD_STEP_DEFINITIONS`](../src/lib/estimate-core/wizard-steps.ts) の並びを維持し、**各ステップの「次へ」条件**と **入力ウィジェット**だけ変える想定（レイアウトパターンは現状維持）。

```text
1. 業種（選択）            → Tier A 必須：未選択なら次へ不可
2. やりたいこと            → 開発の型（選択）必須 ＋ 短い補足（任意・計算に不使用）
3. うまくいっていないこと   → 選択式（計算に不使用）・**任意**（§12.2）
4. 人数（選択）            → Tier A 必須
5. 希望時期（選択）        → Tier A 必須
6. つなぎ（選択）          → Tier A 必須
7. 置き場所のイメージ（選択） → Tier A 必須（`hostingContext`・**つなぎの直後**・実装済み）
8. 使い方・載せる場所       → Tier A 必須
9. 個人情報（選択）        → Tier A 必須
10. 誰が使う（選択）        → Tier A 必須
11. いまの扱い方（選択）   → Tier A 必須
12. 更新頻度（選択）       → Tier A 必須（§5 でステップ自体が出ない場合は §6）
13. デザイン期待（選択）   → Tier A 必須
14. ログイン（選択）       → Tier A 必須（§5 で出ない場合は §6）
15. 予算帯（選択）         → Tier B（任意維持を推奨）
16. 予算の補足            → Tier B
17. 制約・気になること      → Tier B
18. 最終確認（review）
```

## 5. 条件付き非表示ステップとの整合（重要）

[`shouldShowEstimateWizardStepForForm`](../src/lib/estimate/estimate-wizard-step-visibility.ts) により、次のとき **ステップが出ない**。

| 非表示になる質問 | 条件 |
|------------------|------|
| `loginModel` | `dataSensitivity === "no"` かつ（`audienceScope === "internal_only"` **または** `usageSurface === "internal_only"`） |
| `updateFrequency` | `integration === "standalone"` |

**フロー案のルール**

- **非表示の質問は「ユーザーに聞かない」＝コア回答としては除外**し、レンジ計算側では **固定の内部値**を使う（例: `loginModel` は「社内限定・低リスクのデフォルト」、`updateFrequency` は単体運用向けのデフォルト）。
- **採用予定:** 非表示時は **`site-copy` と一致するデフォルト value を `answers` に明示書き込み**し、狭帯判定も **未定扱いにならない**ようにする（§12.2）。

## 6. 「次へ」判定（コア必須化の具体）

- **選択ステップ（Tier A）**  
  - `form.{field} === "unknown"`（または UI 上の「選んでください」相当）のとき **次へ不可**。  
  - 既存の `RadioGroup` の value と `EstimateFormDraft` の初期値を前提にする。
- **`summary`（型選択）**  
  - 型が未選択のとき **次へ不可**。補足テキストの有無は **進行条件にしない**。
- **「わからない／未定」系の選択肢**  
  - ラベルに `わからない` 等が含まれると、現行の `isUnknownAnswer` では **狭帯対象外**になる。  
  - 方針 A: **進行は許すがレンジは広帯**（現状に近い）  
  - 方針 B: **コア軸ではその選択肢を出さない**（必ず具体肢を選ばせる）  
  → **採用予定: 方針 A**（コアは `unknown` で進めないため、未定肢との役割が分かれる）。コピーと `isNarrowRangeEligible` は同期して維持する。

## 7. 最終確認（review）・送信

- `getOrderedAnswerPairs` 等で **Tier A（型含む）がすべて埋まっている**ことをレビュー画面でも一覧表示（既存強化方針と一致）。
- 親ページの送信条件は **`summary` の 8 文字必須を廃止**し、**開発の型が選択済み**かつ **Tier A の残りが満たされている**こと（実装では `canSubmit` を拡張）。

---

## 8. チャット handoff との関係

- `industryBundle` あり：**業種スキップ**は維持。Tier A の残りは必須。
- `path` プリフィルで `teamSize` / `integration` が埋まっている場合：**該当ステップスキップ**は維持し、**必須は満たした扱い**（現行の `prefilledQuestionIds` と同じ思想）。
- トップコンシェルジュの **A3 等**と **開発の型**の対応表があれば、ctx 付き遷移時に **型をプリフィル**できる（任意・§12.2）。

## 9. スコープ外（本メモでは決め打ちしない）

- 選択肢の文言の最終調整（`site-copy.ts` のレビューは別途）。
- **万円レンジの具体値**は **一般的なたたき台**としてコードまたは別表で管理し、**運用で見直す**（方針は §12.1）。本書に固定数値を書き込まない。
- API を **二段生成**に切り替えるかどうか（移行順の推奨は §12.2）。

## 12. 決定の前提・採用予定・実装チェック

ゴールのパイプライン（§2）まで一気通貫させるための **方針の置き場**。以降は **一般的な受託開発・業務システムの目安**をたたき台としてコードや表に落とし、**運用で数値・文言を見直す**前提とする（個別案件の正式見積ではない）。

### 12.1 数値・レンジの扱い（前提）

- **ベースレンジ・係数の具体値は本ドキュメントに固定記載しない。** 実装では **プレースホルダまたはコード内テーブル**に置き、**一般的な相場感**に合わせて更新する。
- 各型・各係数には **「なぜその幅／加算か」1文**をコードコメントまたは表の列で持ち、説明可能性を担保する。
- 表示上は **目安・税別・すり合わせ前提**の免責を既存どおり維持する。

### 12.2 採用予定（推奨案・2026-04 時点）

以下を **実装時のデフォルト**とする（変更時は本節と契約ドキュメントを同期）。

#### 質問・データモデル

| 項目 | 内容 |
|------|------|
| **開発の型** | **`productArchetype`（または同名）を `EstimateFormDraft` に追加**。value は `unknown` 以外の列挙。**`summary` は任意の短い補足テキスト**に専用化（計算に不使用）。answers では型は専用キー、補足は既存「いまいちばん…」または分離キーで **検索しやすく**載せる。 |
| **型の本数** | **6〜8 型**から開始（ダッシュボード／自動化／ナレッジ等。**ラベル・value は実装時に確定**）。 |
| **型ごとのベースレンジ** | 各型 **下限・上限は広め**（規模で桁が変わる型は特に幅大）。**数値は一般的たたき台**としてテーブル化し、根拠列に短文。 |
| **他軸の係数** | **万円の加算**を基本。第1段は **人数帯・つなぎの重さ・外部公開**など **3〜4 本**に限定。**`hostingContext`（閉域寄り・社内のみ等）**は、ルール見積移行時に係数候補として追加しやすい。**合計加算に上限**（例: 一定万円まで）を設け、暴走を防ぐ。乗算は第2段で検討。 |
| **`pain`** | **選択式・任意**（未選択で次へ可）。肢の例: 特にない／属人化・引き継ぎ／連絡・確認の手間／承認・手戻り／情報が散在／品質のばらつき／その他（**5〜7 個目安**。文言は `site-copy` で確定）。 |
| **非表示ステップ** | `loginModel` / `updateFrequency` 非表示時は、**`site-copy` の既存 option value と一致するデフォルト**を選び、**`answers` に明示的に書き込む**（狭帯・説明の一貫性のため）。 |

#### 狭帯・既存ロジック

| 項目 | 内容 |
|------|------|
| **`isNarrowRangeEligible`** | **`summary` の文字数条件は廃止**。代わりに **開発の型が `unknown` でないこと**を条件に含める。リスクブロック・既存8キー相当のチェックは維持（非表示分はデフォルトで満たす）。 |
| **未定系** | **方針 A**（§6）：未定ラベルは進めるが狭帯対象外。コアは **`unknown` で進めない** ので役割が分離される。 |

#### 要件定義・生成

| 項目 | 内容 |
|------|------|
| **テンプレ骨子** | 見出し例: **目的・背景**（型ラベル＋補足）／**スコープのたたき台**（選択の表または箇条書き）／**前提・未確定**／**次のすり合わせ**。機械差し込みを厚くし、LLM は文体・フォローアップに寄せる。 |
| **プロンプト** | **数値レンジは入力された値のみ。補足テキストで変更禁止**をシステム側に明記。 |
| **移行順** | **第1段**: ルール（または固定ロジック）で万円レンジを決定し、**既存一括 `generateObject` に数値を固定渡し**（スキーマは当面維持可）。**第2段**で生成を分割する。 |

#### handoff・契約・回帰・§1.1 コピー

| 項目 | 内容 |
|------|------|
| **契約ドキュメント** | `industry-estimate-handoff-contract.md`・`estimate-sample-output-contract.md` に **型キー・pain 任意**・handoff 時の型プリフィルを追記。 |
| **ゴールデン** | 最低 **3 本**: (1) 一般・小規模・狭帯が理論上あり得る (2) 士業・大規模・狭帯ブロック (3) 単体運用で `updateFrequency` 非表示。 |
| **本見積導線（固定文）** | **ウィザード冒頭** 約2行／**結果ページ** 約1行／**問い合わせ CTA 直上** 約2行（本見積・詳細は担当すり合わせ／別フォーム想定）。文言は `estimateDetailedCopy` 等で管理。 |

## 14. 問い合わせページとの統合（実装メモ）

`/contact` では、詳細見積ウィザードと**同一の質問ID・ステップ順**（`ESTIMATE_WIZARD_STEP_DEFINITIONS` / `shouldShowEstimateWizardStepForForm`）でヒアリングし、直流入でも送信前に `inquiryBrief` を必須とする。

| 内容 | 主なファイル |
|------|----------------|
| 質問の単一ソース | `src/lib/estimate-core/question-model.ts`・`wizard-steps.ts` |
| 直流入ヒアリング | `src/components/contact/ContactIntakeHearingBlock.tsx`（`EstimateDetailedHearingWizard` を再利用） |
| 見積回答の復元・ロック | `src/lib/contact/contact-intake-context.ts`・`parse-estimate-answers-to-form-draft.ts` |
| 問い合わせのみのスナップショット（`inquiry-brief` 用） | `src/lib/contact/build-contact-synthetic-snapshot.ts` |
| 送信ゲート | `EstimateDetailedAmountContent` と同様に `readiness` または必須フォローアップ完了 |

詳細見積フロー本体（`EstimateDetailedFormContent` / `EstimateDetailedAmountContent`）は変更しない。問い合わせ側が同じウィザードと API を読むだけに留める。
