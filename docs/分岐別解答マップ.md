# AIコンシェルジュ 分岐別解答マップ

## 目的
- どの分岐でどの解答が出るかを可視化する。
- 「寄り添い」と「次の行動」まで一貫した体験にする。
- 更新時に、回答欠落や導線欠落を防ぐ。

---

## ルート分岐（ROOT_CHOICES）

| root optionId | 表示文言 | 次ステップ |
|---|---|---|
| `root_a` | 開発コストの概算を知りたい | `A3` → `A_SCOPE` → `buildEstimateBlockA` |
| `root_b` | コンサル・伴走の概算を知りたい | `B2` → `B_SCOPE` → `buildEstimateBlockB` |
| `root_cde` | 技術・ツール・進め方を知りたい | `CDE_PICK` → `C2 or D2 or E2` → `buildCdeSummaryBlock` |
| `root_e` | まず相談・窓口の進め方を知りたい | `E2` → `buildCdeSummaryBlock("E")` |

## Aトラック（開発コスト）

### 質問
- `A3` いま作りたいイメージ
- `A_SCOPE` 規模と連携イメージ

### ひとことの整理（`buildEmpathyLineA`）

| optionId (`A3`) | 出力テキスト |
|---|---|
| `build_poc` | 小さく試す前提なら、まずは効果検証に集中できる進め方が合います。 |
| `build_auto` | 既存業務の自動化は、優先順位を付けて段階的に進めると扱いやすくなります。 |
| `build_chatbot` | 社内チャットボットは、まず利用シーンを絞ると定着しやすいです。 |
| `build_inquiry` | 問い合わせ・受付の自動化は、例外処理の設計が品質に直結します。 |
| `build_platform` | 部門横断の基盤は、最初の範囲設計が後工程の手戻りを抑えます。 |
| `build_other` / 未知 | 今回の条件だと、まずは小さく効果を確認しながら進めるのが安全です。 |

> `freeform` がある場合は末尾に補足が付きます。

## 共通「次の行動」文（`buildNextActionLines`）
- まずは体験demoで操作感を掴む
- 詳細見積もりで要件のたたき台をつくる

---

## 出力順ルール
- A/B: 概算レンジ → ひとことの整理 → 前提条件 → 次の一歩 → 選択内容（参考） → 免責
- C/D/E: ご案内 → ひとことの整理 → 次の一歩 → 選択内容（参考） → 免責

---

## UI導線（結果画面）

- 結果本文内の「次の一歩」テキストは、表示時にスクロール領域から分離される。
- 結果カード直下に、2カラムの導線ボタンを表示する。
  - 左: `demoを体験`（分岐に応じて `/experience/{slug}` または `/demo/list`）
  - 右: `要件をまとめてみる`（`/estimate-detailed` へコンテキスト付き遷移）
- 「次の一歩」という見出し文言は、UI上には表示しない。
