# インタラクティブ体験デモ — 方向性・要件まとめ

最終更新: 2026-03-26

## 1. このドキュメントの位置づけ

- **目的**: 社内ナレッジBOT・飲食ダッシュボード以外の **インタラクティブ体験**（`/experience/{slug}`）を、モック中心で **体験価値とUIギミックまで作り込む** ための共通の北極星にする。
- **実装の単一ソース（ルーティング・一覧）**: [`src/lib/experience/prototype-registry.ts`](../src/lib/experience/prototype-registry.ts)
- **各体験の実体**: [`src/components/experience/prototypes/*.tsx`](../src/components/experience/prototypes/)
- **文章デモ（Sanity）との対応**: 各プロトタイプの `demoSlug` は [`docs/demo-mock-inventory.md`](demo-mock-inventory.md) のカタログと揃える。

「完成済み（参照用）」として触れない詳細要件に縛られないが、トーンと情報設計は揃える:

| slug | タイトル（略） | 備考 |
|------|----------------|------|
| `internal-knowledge-share-bot` | 社内ナレッジ共有BOT | 注力・完成 |
| `restaurant-ops-dashboard-demo` | 飲食オペレーション・ダッシュボード | 注力・完成 |

以下 **③〜⑩** が本ドキュメントでの強化対象（既にコンポーネントあり → **差分強化** が主作業）。

## 2. 全体方針（8本共通）

### 2.1 技術・品質

- **モック優先**: 決定論的なダミーデータ、段階的な `setTimeout` / ストリーミング風UI、状態マシンで十分。本番APIは不要。
- **アクセシビリティ**: キーボード操作、フォーカス、`prefers-reduced-motion` で派手なアニメを抑える（プロジェクトルール準拠）。
- **Tailwind**: 本文の 1rem 相当は `text-[1rem]` 等（`text-base` は文字色衝突のため避ける）。

### 2.2 「テンプレではなく別アプリ感」

- **共通化してよいもの**: フォーカスリング、余白の段階、セマンティクス、エラー時の見せ方。
- **デモごとに変えるもの**: ヘッダー／サイドバーの比喩、1プライマリカラー、角丸・濃度、主要レイアウト（メール型・チケット型・スマホ枠型など）。
- **推奨**: 薄い **Experience Chrome**（タイトルバー＋テーマクラス）を props で切替え、**メインの「出力エリア」はデモ専用** で実装する。

### 2.3 コードだけで難しいもの・素材が効くもの

| 種別 | 方針 |
|------|------|
| 検索ログ・照合ログ・タグ付与アニメ | **コードのみで可**（逐次表示、Framer Motion 等）。 |
| スライダー連動グラフ | **コードのみで可**（range + SVG）。手はかかるが動画不要。 |
| 疑似音声波形 | **本物のマイク不要**。CSS/Canvas のループアニメ＋タイプライターでモック。 |
| 現場写真・領収書 | **静止画サンプル**（`public/`）があると没入感が上がる。動画は必須ではない。 |

## 4. 実装を進める単位の提案

### 4.1 一つの MD で8要件を載せる意味

- **要件の抜け・ブレ** を防ぎ、デザインの「別アプリ感」と「共通スパイン」のバランスを共有できる。
- レビュー時に **このファイルを参照** すれば、個別PLANが短くて済む。

### 4.2 「一気に8つ実装」について

- **可能ではある** が、変更量が **8ファイル＋共通部品の抽出** になり、1PRだとレビュー負荷とリグレッションリスクが高い。
- **推奨**:  
  - **Phase A（低リスク・高体感）**: ④⑤⑩ — 状態とレイアウト中心。**実装済（2026-03-26）** — モック・UI 強化。  
  - **Phase B（ログ・ストーリー）**: ③⑧ — 実装済（2026-03-26）: ログ表示、統合回答/参照元リンク、規定照合ログ/違反時アドバイス。
  - **Phase C（インタラクション重め）**: ⑥⑨ — 実装済（2026-03-26）: マジック・スライダー（SVGグラフ/トピックス）＋波形→書き起こし変換のタイピング（モック）。
  - **Phase D（画像・枠）**: ⑦ — 実装済（2026-03-26）: 縦長スマホ枠・画像内ピン・修繕依頼文案パネル・`public/media/experience` サンプルSVG。

並行作業する場合は **Phase ごとにブランチ**、または **デモ2本ずつPR** が現実的。

### 4.3 個別実装PLANを切るタイミング

- 各デモで **画面状態（初期 / 処理中 / 完了）** のスクショ1枚ずつ、または箇条書きで固めたら、**そのデモ専用の短いPLAN**（チェックリスト形式）を切ると実装が速い。
- 共通Chromeやログコンポーネントを **最初の1本で試作** し、③⑧で再利用するのが効率的。

## 5. 完了の定義（各デモ共通チェックリスト）

実装完了とみなす最低ライン:

- [ ] 本節の **入力 / 出力 / UIギミック** がユーザー操作1往復で確認できる。
- [ ] `prefers-reduced-motion` で主要アニメが過剰にならない。
- [ ] キーボードで主要操作が可能（ボタン・スライダー・フォーカス移動）。
- [ ] `npm run build` が通る。
- [ ] 新規デモを増やした場合は [`docs/demo-mock-inventory.md`](demo-mock-inventory.md) のルールに従い更新（本リファクタは既存 slug の強化が主）。
