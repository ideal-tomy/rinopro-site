# パフォーマンス改善ガードレール（見た目維持前提）

## 目的
- 演出の印象を大きく変えずに、初期JS・hydrate・main-thread負荷を抑える
- モバイルを優先しつつ、デスクトップの体験品質を維持する

## 実装ルール
- **共通殻に重い処理を直置きしない**  
  `layout` 配下は軽量化を優先し、重い UI は遅延ロードを検討する。
- **client 化は必要最小限**  
  相互作用がない UI は server component 側に寄せる。
- **モバイルの演出は粒度を粗くする**  
  1要素ごとに追従するスクロール演出より、セクション単位の演出を優先する。
- **blur と無限アニメは限定利用**  
  `filter: blur(...)` と常時ループは、モバイルで負荷を確認してから採用する。

## 計測運用
- 拡張機能なしの同条件で、`/`, `/demo`, `/services`, `/about`, `/contact` を比較する
- 記録する指標: `performance`, `LCP`, `TBT`, `interactive`, `main-thread work`
- 変更ごとに差分を残し、悪化した場合はその変更単位で戻せるようにする

## 退行を防ぐチェック観点
- チャット未操作でチャット本体が読み込まれていないか
- `/services` の未表示タブの詳細が初回から読み込まれていないか
- モバイルで `ScrollSequence` が過剰に動作していないか
- 背景演出（Three.js）の負荷設定がモバイル向けに抑制されているか
