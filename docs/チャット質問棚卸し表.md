# チャット質問棚卸し表

## 目的

- 各チャットボットが**どの段階の言語化を担当しているか**を明確にする。
- 「同じことを別の言い方で聞いていないか」「問い合わせへ送るのが早すぎないか」を棚卸しする。
- [`チャット共通facts一覧.md`](./チャット共通facts一覧.md) の canonical facts を、どの導線が担当しているかを見える化する。

---

## スコープ

本書で扱うのは、**サイト全体のコンシェルジュ〜見積〜問い合わせ**に関わる導線。

- トップの AI コンシェルジュ
- `/demo/list` の選択式コンシェルジュ
- サービスカード経由チャット
- グローバル freeform チャット
- 詳細見積ウィザード
- 見積後の問い合わせ前整理
- 問い合わせ直流入のヒアリング

体験プロトタイプ内の独立 bot は、**サイト全体の受け皿設計**とは役割が異なるため本書では主対象外。

---

## 全体ストーリー

理想のストーリーは次の流れ。

1. **トップ / demo / services**
   - 欲しいものを言語化
   - 近い成果物イメージを持ってもらう
   - 次に demo / 見積 / freeform のどれが適切かを案内
2. **詳細見積**
   - 要件定義と概算に効く facts を structured に揃える
   - 仮要件定義と概算レンジを作る
3. **問い合わせ前整理**
   - 今回の返信で何を求めているかを明確にする
   - 初回返信をテンプレ中心で返せる状態まで整える
4. **問い合わせ**
   - 連絡先と最終補足を送る

## Phase 3 の見方

- 質問定義の正規レイヤーは `src/lib/chat/question-definition.ts` を基準にし、`questionId -> optionId -> emitsFacts` の順で見る。
- トップは `src/lib/chat/concierge-flow-definitions.ts` の `TOP_FLOW_STEP_DEFS`、demo は `src/lib/demo/intelligent-concierge.ts` の `DEMO_CONCIERGE_QUESTION_DEFS`、services は `src/lib/chat/service-card-preset-content.ts` の `getServiceStep1Defs()` / `getStep2Defs()` を見る。
- `optionId` は表示ラベルと分離した安定 ID とし、handoff・routing・analytics では原則こちらを参照する。
- `emitsFacts` は canonical facts を即確定するだけでなく、`candidate` / `approx` を含めて「この質問が意味的に何を出すか」を表す。

### 導線ごとの `emitsFacts` の重心

| 導線 | 主な `optionId` 粒度 | 主な `emitsFacts` | 備考 |
|------|----------------------|-------------------|------|
| トップ AI コンシェルジュ | `root_need_touchpoint`, `build_auto`, `scope_m_flex` など | `productCategory`, `productArchetype`, `teamSize`, `integration`, `entryIntent` | A/B は `approx` を多く含む |
| demo一覧コンシェルジュ | `construction`, `management`, `reporting`, `full_auto` など | `industryBundle`, `currentPain`, `productCategory`, `productArchetype` 候補 | 役割軸は `targetSummary` 候補として扱う |
| サービスカード経由チャット | `dev_what`, `期間の目安を知りたい` など | `entryIntent`, `desiredReply`, `productCategory`, `timeline` 候補 | 説明導線だが意味モデルは共通化する |
| 詳細見積ウィザード | `summary`, `timeline`, `integration` など | `ESTIMATE_QUESTION_FACT_MAP` に準拠 | 共通契約の参照実装 |

## 現状の質問構造の要約

### 1. トップ AI コンシェルジュ

Phase 7 以降の入口は、**知りたい情報**ではなく**欲しい成果物・状態**を起点にしている。

```30:36:src/lib/chat/concierge-flow-definitions.ts
export const ROOT_CHOICES: FlowChoice[] = [
  { id: "root_need_touchpoint", label: "お客様との接点を作りたい" },
  { id: "root_need_management", label: "顧客・案件管理を楽にしたい" },
  { id: "root_need_automation", label: "社内作業を減らしたい" },
  { id: "root_need_reception", label: "予約・受付を自動化したい" },
  { id: "root_need_new_service", label: "独自サービスを形にしたい" },
  { id: "root_need_clarify", label: "まだうまく言えないので整理したい" },
];
```

#### 向いている役割

- 金額感・方向性の短時間整理
- 大まかな温度感の判定
- 詳細見積や demo への橋渡し

#### 弱い点

- `productCategory` / `productArchetype` を canonical に取りに行くには、まだ入口が費用軸寄り
- `依頼方法` トラックは軽い相談を通しやすい文言が残っている

#### 理想の担当

- `欲しいもの`
- `どこまでやりたいか`
- `次に demo を見るべきか、見積に進むべきか`

### 2. demo一覧コンシェルジュ

`/demo/list` は、業種 × 役割 × 課題 × 自動化深度で demo を決定論的に絞る構造になっている。

```27:35:src/lib/demo/intelligent-concierge.ts
export type ConciergeAnswers = {
  domain: ConciergeDomainId;
  domainDetailId?: string | null;
  domainNote?: string | null;
  audienceRole: AiDemoAudienceRole;
  issue: AiDemoIssueTag;
  automationDepth: AiDemoAutomationDepth;
};
```

#### 向いている役割

- 近い成果物イメージを持たせる
- 業界と課題の輪郭を言語化する
- `industryBundle` と `currentPain` を早く取る

#### 弱い点

- `timeline`, `integration`, `usageSurface` など見積に効く facts は取っていない
- demo の振り分けには強いが、問い合わせ readiness の判定にはまだ遠い

#### 理想の担当

- `industryBundle`
- `audienceRole`
- `currentPain`
- `productCategory` の候補化

### 3. サービスカード経由チャット

サービスカードの質問は、「知りたいこと」を短く選ばせる構造。

```17:25:src/lib/chat/service-card-preset-content.ts

  { key: "con_support", label: "支援内容・進め方を知りたい" },
  { key: "con_outcome", label: "成果・費用感を知りたい" },

```

#### 向いている役割

- 開発と伴走の違いを理解させる
- 何を知りたいかの論点整理

#### 弱い点

- 案件 facts はほぼ取らない
- 問い合わせへ送るには情報が薄い

#### 理想の担当

- `entryIntent`
- 開発寄り / コンサル寄りの方向性確認
- 見積か freeform へ送る分岐

### 4. グローバル freeform チャット

ここは structured choice で拾えない曖昧要望の受け皿。

#### 現状の役割

- 下書き注入
- visitor journey への自由記述蓄積
- demo ハブではテキストからおすすめ 3 件の推定

#### 理想の役割

- `freeformMemo` の主要入力口
- 音声入力の受け皿
- AI による言語化支援の本丸

#### 前提

- ここは「freeform を減らす」のではなく、「freeform を facts に変換する設計」が必要

### 5. 詳細見積ウィザード

詳細見積は、現状最も structured な facts 収集フロー。

#### 向いている役割

- 仮要件定義
- 概算レンジ
- 見積と要件に効く中核 facts の確定

#### 弱い点

- `summary` が 8 文字ルールに依存
- `productArchetype` と `problemSummary` がまだ分離し切っていない

#### 理想の担当

- 見積に効く facts を structured に揃える
- 文字数ガードではなく、必要 facts 充足で進行させる

### 6. 見積後の問い合わせ前整理

- `inquiryIntent`
- `desiredReply`

### 7. 問い合わせ直流入

問い合わせページは、現状でも「準備済み文脈がないと送信フォームが出ない」構造。

- 直流入でも最低限の案件具体化を揃える
- 見積由来文脈をロックして不足分だけ聞く

#### 弱い点

- 見積直後と直流入で、まだ心理的な一貫性が弱い
- `additionalNote` 以外の最終本気度質問は未分離

#### 理想の担当

- 連絡先収集
- 最終送信判断
- 本気度を見る最後の補足 1 問

## 重複しやすい質問の棚卸し

| テーマ | 現状の現れ方 | 重複リスク | 今後の方針 |
|--------|--------------|------------|------------|
| 何を作りたいか | トップ `A3`、見積 `summary`、問い合わせ `problemStatement` | 高い | `productArchetype` と `problemSummary` を分ける |
| 誰のどの業務か | demo の `audienceRole`、問い合わせ `targetSummary`、見積の `audienceScope` | 中〜高 | `targetSummary` を canonical にし、demo は候補化に使う |
| いつまでに判断したいか | 見積 `timeline`、問い合わせ `decisionTimeline` | 中 | 見積は開発時期、問い合わせは意思決定時期として整理しつつ相互変換 |
| 何が困っているか | demo `issue`、見積 `pain`、問い合わせ `problemStatement` | 高い | `currentPain` と `problemSummary` を区別 |
| 相談でほしいもの | サービス preset、問い合わせ `inquiryIntent` / `desiredReply` | 中 | 問い合わせ直前で canonical に確定 |

## 理想の担当分担

| 段階 | 主担当導線 | 取るべき facts | 備考 |
|------|------------|----------------|------|
| 言語化前 | トップ / freeform | `productCategory`, `freeformMemo` | AI による言語化支援を強く使う段階 |
| イメージ形成 | demo一覧 | `industryBundle`, `audienceRole`, `currentPain` | 近い成果物を見ることで理解を進める |
| 方向決め | services / トップ完了後 | `entryIntent`, `desiredDirection` | 開発 / コンサル / 見積 / demo の振り分け |
| structured 化 | 詳細見積 | 見積 facts 一式 | 最も重い structured 質問はここで担当 |
| 送信判断 | 問い合わせ前整理 / 問い合わせ | `inquiryIntent`, `desiredReply`, `decisionTimeline`, `whyNow` | 初回返信可能な状態まで詰める |

## Phase 1 で固定した owner / state の見方

- owner は `visitorJourney` / `estimateFormDraft` / `inquiryBrief` の 3 つに固定する。
- state は `direct` / `approx` / `candidate` / `missing` を使い、導線ごとに「確定」「近似」「要確認」を切り分ける。
- `productArchetype` と `problemSummary` は現状まだ `summary` に同居しているため、詳細見積ではいったん candidate 扱いを許容し、後続フェーズで質問分離する。
- `timeline` は開発時期、`decisionTimeline` は意思決定時期として別管理し、問い合わせ前整理で接続する。

### 導線ごとの owner への載せ方

| 導線 | まず載る owner | 主な state | 備考 |
|------|----------------|------------|------|
| トップ AI コンシェルジュ | `visitorJourney` | `approx` / `candidate` | freeform は `freeformMemo` として保持 |
| demo一覧コンシェルジュ | `visitorJourney` | `direct` / `approx` | `industryBundle` と `currentPain` の候補化が中心 |
| サービスカード経由チャット | `visitorJourney` | `candidate` | 入口意図の補助が主 |
| グローバル freeform チャット | `visitorJourney` | `candidate` | AI による言語化支援の受け皿 |
| 詳細見積ウィザード | `estimateFormDraft` | `direct` | `summary` 系だけは Phase 1 では candidate を許容 |
| 見積後の問い合わせ前整理 | `inquiryBrief` | `direct` | `inquiryIntent` と `targetSummary` を canonical 化 |
| 問い合わせ直流入 | `estimateFormDraft` -> `inquiryBrief` | `direct` | 最低 facts を埋めてから送信判断へ進める |

## この棚卸し表から出る次アクション

### 優先度 高

1. トップの入口を「知りたい情報」起点から「欲しいもの」起点へ寄せる
2. `productArchetype` と `problemSummary` を分離する
3. 問い合わせ直前に「本気度を測る最後の自由記述 1 問」を設計する

### 優先度 中

1. demo一覧の回答を `productCategory` 候補へ翻訳する
2. サービスカード導線を「説明用」から「次の一歩の振り分け用」へ寄せる
3. freeform から facts を抜くルール / AI 補助の設計を明文化する

### 優先度 低

1. すべての導線で文言を完全統一する
2. すべての自由記述を即時 AI 判定にする
