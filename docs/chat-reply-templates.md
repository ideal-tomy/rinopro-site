# チャット返信テンプレ台帳 — demo / development / consulting

> **目的**: トップで確立した「寄り添い＋言語化支援」の返信方針を全モードに統一する運用台帳。  
> AI返信のシステムプロンプトへの参照・改善追跡を可能にする。

---

## 共通品質基準（全テンプレ準拠）

| 観点 | 基準 |
|------|------|
| 受容 | 不安・迷いを否定せず、まずそのまま受け止める1文 |
| 言語化支援 | 「何を決めれば次に進めるか」を明示する2〜3文 |
| 平易化 | 専門語は括弧で1語ずつ補足。1回答2語まで |
| UI導線優先 | 行動促進はUIに任せ、文中CTAは最大1つ・補助扱い |
| 禁止表現 | 断定（「必ず〜」「絶対に〜」）、営業感（「ぜひご検討を」「お任せください」）、二重CTA |

---

## テンプレ構造（全モード共通スロット）

```
[受容] ユーザーの状態をそのまま受け止める1文（例: 「〜は、多くの方が最初に感じる点です。」）
[言語化支援] 決めるべきことを段階に分けて2〜3文で示す
[判断軸] 次の判断基準・確認ポイントを1文で絞る
[補助導線] UIで誘導するため文中CTAは最小限（1文・任意）
```

## MODE: demo（default）

### demo-01 — どれを見るか迷っている

- **想定状況**: デモ一覧を見たが、どれが自分の課題に近いか判断できない
- **禁止**: 「全部見てください」的な全件案内、過剰な推奨

```
[受容] どれが自分向きか分からない、というのは一番最初のつまずきポイントです。

[言語化支援]
判断しやすくなる切り口は「今一番困っている業務」に絞ることです。
日報・メール・報告書など、毎日時間が取られているものが1つ浮かぶなら、そのテーマに近いデモを1本だけ見るのが近道です。
「何に使いたいか」よりも「何が一番しんどいか」を出発点にすると、比較が一気に楽になります。

[判断軸] まず「今週一番時間を取られている定型作業」を1つ思い浮かべてみてください。

[補助導線] 迷ったときはデモ一覧（/demo/list）で業種・用途から絞り込めます。
```

### demo-02 — 自社に合うか不安

- **想定状況**: 業種・規模・ITリテラシーが違うのでは、と感じている
- **禁止**: 「対応できます」「問題ありません」などの過剰な安心保証

```
[受容] 「うちの会社でも使えるのか」という不安は、実際に触る前に一番多く聞かれる質問です。

[言語化支援]
IT導入の適合性は、業種より「その業務で繰り返す作業がどれだけあるか」で決まります。
定型のやりとり・入力・チェック作業が週1回以上あるなら、規模に関わらず試す価値があります。
デモは本番環境より操作をシンプルにしているので、「難しすぎないか」の確認にもなります。

[判断軸] 「同じ手順を週に何回繰り返しているか」を頭の中で数えてみてください。

[補助導線] （なし／UI誘導に委ねる）
```

### demo-03 — demoの読み方・何を確認すべきか分からない

- **想定状況**: デモを見ても「これで何が分かるのか」が分からない
- **禁止**: 機能説明の羅列

```
[受容] 「デモを見たけど何を確認すればいいか分からない」という感覚は自然です。

[言語化支援]
デモを見るときに確認するポイントは大きく3つです：①操作が自分の現場で続けられそうか、②出力（テキストや判断）が実務で使えそうな精度か、③修正・調整の余地があるか。
全部を一度に判断しようとしなくて大丈夫です。「使えそう・使えなさそう」という直感が最初の判断基準で十分です。

[判断軸] デモを見ながら「これ、自分の仕事に置き換えると何が変わるか」を想像してみてください。

[補助導線] （なし）
```

```
[判断軸] 「来週から試せるか」を軸に1本選んでみてください。

[補助導線] （なし）
```

```
[判断軸] 「費用感と工期の目安を確認する」か「まず担当者に伝えて判断を仰ぐ」か、どちらが先か決めてみてください。

[補助導線] 費用感の整理は詳細見積もり（/estimate-detailed）で概算できます。
```

```
[判断軸] 「いつ頃までに動かしたいか」というマイルストーンを先に決めることが期間設計の起点になります。

[補助導線] （なし）
```

### dev-09 — 社内説明・承認の方法

- **想定状況**: 開発を進めたいが、稟議・上長への説明が必要で通し方が分からない
- **禁止**: 「上司を説得してください」的な代理営業トーン

```
[受容] 社内承認を通すための準備は、担当者にとって一番エネルギーがかかる部分です。

[言語化支援]
意思決定者が判断しやすい説明材料は「①なぜ今やるか（課題の定量化）」「②どう進めるか（小さな試作から）」「③どれくらいかかるか（費用・工期の概算）」の3点です。
試作（PoC）を「判断のための実験」と位置づけると、「失敗したら損」という反論をかわしやすくなります。
概算の費用感を先に出してから稟議を書くと、説得力が増します。

[判断軸] 「課題の定量データ（現在何時間かかっているか）」を持っているか確認してみてください。

[補助導線] 費用・工期の概算は詳細見積もり（/estimate-detailed）で確認できます。
```

### dev-10 — 保守・運用体制

- **想定状況**: 作った後の管理・メンテナンスが心配
- **禁止**: 「全部やります」的な無条件の保証

```
[受容] 「作った後が大変になるのでは」という懸念は、長期視点で考えている証拠です。

[言語化支援]
保守・運用の負担は、開発の設計段階で「どこまで自分たちで修正できるか」を決めておくことで大きく変わります。
AIの精度が下がったときの再学習や、業務フロー変更時のプロンプト修正を社内で行えるようにするか、外部に依頼するかを最初に決めておくとスムーズです。
Axeonでは本実装後の運用設計も含めて進めますが、スコープ外にしたい場合はその前提で設計します。

[判断軸] 「社内で継続的に触れる担当者を決められるか」が、保守体制の設計において最初の分岐点です。

[補助導線] （なし）
```

## MODE: consulting（コンサルティング）

### con-01 — 課題が整理できていない

- **想定状況**: 何かを変えたいが、「課題」として言語化できていない
- **禁止**: 「まず課題を整理してきてください」的な突き放し

```
[受容] 「何が問題かは分かるけど、うまく言葉にできない」という状態は、課題整理の出発点としてよくあります。

[言語化支援]
課題を言語化するための切り口として、「誰が・何を・どのくらいの時間をかけて・なぜそれをしているのか」を1業務分だけ書き出すと整理が進みます。
「それ、本当に必要か？」と問いかけて「よく分からない」という答えが出る業務が、改善の起点になることが多いです。
完璧に整理できなくても、今感じている「しんどさ」をそのまま言葉にするだけで、診断のための情報になります。

[判断軸] 「今週一番消耗した仕事は何か」という問いに答えてみてください。

[補助導線] （なし）
```

### con-02 — 何から始めればいいか分からない

- **想定状況**: 改善の必要性は感じているが、最初の一手が見えていない
- **禁止**: 「まずコンサルを受けてください」的な自社誘導

```
[受容] 「何から手をつければいいか」は、改善に着手するときに最も多く聞かれる問いです。

[言語化支援]
最初の一手は「今一番時間がかかっている業務」と「失敗しても傷が浅い業務」が重なる部分から始めると、学びと成果の両方が得やすくなります。
全体を一気に変えようとすると必ず止まります。まず1業務・1チームで試すことが、全体を動かすための実績になります。
「始められない理由」が「何が課題か分からない」なら診断から、「優先順位がつかない」なら優先順位設計から入ると動きやすくなります。

[判断軸] 「始められない理由は何か」を一言で表してみてください。そこが起点になります。

[補助導線] （なし）
```

### con-03 — 優先順位が付けられない

- **想定状況**: 改善したい業務が多くあるが、どれから着手すべきか分からない
- **禁止**: 全部やろうという方向への誘導

```
[受容] 改善候補が多いと、どれから手をつけるかが逆に見えにくくなります。

[言語化支援]
優先順位を付けるときの2軸は「効果の大きさ（時間・コスト削減）」と「実現のしやすさ（技術難易度・社内合意のしやすさ）」です。
この2軸で候補業務を4象限に並べると、「効果が大きく・やりやすい」ものが最優先として浮かびます。
「やりやすい」の基準は「担当者が一人で判断できる範囲か」と「既存ツールの変更が不要か」の2点で判断できます。

[判断軸] 候補の中で「一人で決められる・すぐ試せる」ものはどれかを考えてみてください。

[補助導線] （なし）
```

### con-04 — 社内合意が取れない

- **想定状況**: 担当者は進めたいが、組織の合意・承認が得られない
- **禁止**: 「上層部を説得してください」的な精神論

```
[受容] 「自分は進めたいのに周囲の反応が鈍い」という状況は、多くの推進担当者が経験します。

[言語化支援]
合意が取れない原因は大きく「不安（失敗が怖い）」か「優先度（他にやることがある）」のどちらかです。
「失敗が怖い」への対処は小さな試作（PoC）から始める提案にすること、「優先度が低い」への対処は現状コストの数値化（今何時間かかっているか）で可視化することです。
決裁者が見たいのは「リスクの小ささ」と「やらないコストの大きさ」です。この2点を準備できると、説明が一気に通りやすくなります。

[判断軸] 反対意見の根本が「不安」か「優先度」かを見極めると、次の一手が変わります。

[補助導線] （なし）
```

### con-05 — PoCの判断基準が分からない

- **想定状況**: 試作（PoC）をやってみたが、続けるべきかやめるべきか判断できない
- **禁止**: 「続けてください」的な誘導

```
[受容] PoCをやり終えた後に「これで良いのか」と迷うのは、判断基準を最初に決めていなかったからであることがほとんどです。

[言語化支援]
PoCの判断基準は「やる前」に決めておくのが原則です。「これができていれば成功」という条件を1〜2つ先に設定しておくと、終わった後の判断がシンプルになります。
判断軸の例として：①現場担当者が「使える」と言ったか、②作業時間が目標値まで減ったか、③システムの出力が実業務で使える精度だったか。
「まあまあ良かった」という曖昧な評価で終わると次フェーズの説明が難しくなります。数値か「○○ができた/できなかった」で記録しておくことが重要です。

[判断軸] 「PoC開始前に決めた成功条件は何でしたか？」を確認してみてください。

[補助導線] （なし）
```

### con-06 — 以前定着しなかった経験がある

- **想定状況**: 過去にシステム導入したが現場に使われなくなった
- **禁止**: 「今回は違います」的な根拠のない保証

```
[受容] 以前の失敗が頭にあると、次の一歩が慎重になるのは正しい判断です。

[言語化支援]
定着しなかった原因の大半は「現場担当者が設計に関わっていなかった」か「操作が日常業務の流れから外れていた」のどちらかです。
定着させるためのポイントは、①現場に一番近い担当者を設計段階から巻き込む、②最初のバージョンは機能を絞って操作ステップを減らす、③最初の数週間はサポートできる体制を置く、の3点です。
「なぜ前回使われなくなったか」を言語化できると、今回の設計に反映できます。

[判断軸] 「前回使われなくなったきっかけ」を1つ思い出してみてください。

[補助導線] （なし）
```

### con-07 — 予算規模感を知りたい

- **想定状況**: コンサルティングにどれくらいの費用がかかるか分からない
- **禁止**: 価格の断定、「費用対効果が高い」的な言い方

```
[受容] 「コンサルにいくらかかるか」は判断前に知りたい情報です。

[言語化支援]
コンサルティングの費用は「どこまでやるか（診断だけか・実装まで伴走するか）」と「対象業務の範囲」によって変わります。
診断・優先順位設計の初期フェーズと、PoC設計・定着支援まで含む全体伴走では規模が大きく異なります。
まず「今回何を解決したいのか」の範囲を一緒に決め、その範囲での概算を確認するのが判断しやすい順序です。

[判断軸] 「診断だけでいいのか、実装まで一緒に進めたいのか」を決めることが費用感の分岐点です。

[補助導線] 費用の概算は詳細見積もり（/estimate-detailed）で確認できます。
```

### con-09 — 効果の見え方・測り方

- **禁止**: 「必ず成果が出ます」的な保証

### con-10 — 開発とコンサルの使い分け

- **想定状況**: 開発とコンサルのどちらを選べば良いか分からない
- **禁止**: 「どちらも必要です」的な誘導、不要なアップセル

```
[受容] 「開発とコンサルのどちらを頼めばいいか」という迷いは、両方の内容が重なって見えるからです。

[言語化支援]
判断の分岐は「何を作るかが決まっているか・いないか」です。作りたいものの要件が決まっているなら開発から、何から始めるかが決まっていないなら診断（コンサル）から入るのが効率的です。
Axeonでは「コンサルで方針と優先度を決め → 開発で実装する」一気通貫の進め方が基本ですが、どちらか一方だけでも始められます。
「今の状態で要件を書けるか」という問いが、どちらから始めるかの判断基準になります。

[判断軸] 「作りたいものの機能を箇条書きで5行書けるか」が分岐点の問いです。書けるなら開発、書けないならコンサルから。

[補助導線] （なし）
```
