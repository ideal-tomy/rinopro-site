# 問い合わせ改善PLAN

## 目的

- [`問い合わせポリシー.md`](./問い合わせポリシー.md) を、チャット・詳細見積・問い合わせ導線へ段階的に落とし込む。
- 「リテラシーが低くても使いやすいが、本気度の低い問い合わせは通しにくい」状態を、UI ではなく**導線設計と送信ゲート**で実現する。
- 初回返信の運用コストを下げつつ、本気度の高い案件だけを次段階へ進められるようにする。
- 全体の親計画は [`チャット刷新実行計画.md`](./チャット刷新実行計画.md) とし、本書はその中でも問い合わせに関わる実装計画を扱う。

---

## ゴール像

### ユーザー体験

- トップや demo から来ても、最初に「欲しいもの」の整理から入れる
- 同じ意味の質問は、導線をまたいで繰り返されない
- 問い合わせ前に、自動で仮要件定義と概算レンジが見える
- 送信時には「この前提で相談する」という意思決定が伴う

### 事業運用

- 初回返信はテンプレ中心で返せる
- 情報収集目的・比較目的の軽い問い合わせは demo / 詳細見積で止まりやすい
- 2 通目や 30 分相談に進む相手の密度が上がる

---

## 今回の PLAN が扱う範囲

- チャット入口の分類軸
- 問い合わせへ進める判断基準
- 詳細見積の送信ゲート
- 問い合わせ前の最終確認項目
- 文言・CTA・導線の役割分担

## 今回の PLAN ではまだ固定しないもの

- 金額レンジの具体的な相場テーブル
- 自動生成プロンプトの細部
- メール返信テンプレ本文の最終文面
- demo ルーティングの個別 slug マップ

---

## 設計方針

### 1. 導線ごとに質問は変えてよい

- すべての入口で同じ質問列を使う必要はない
- ただし内部で保存する facts は共通化する

### 2. 引き継ぐ単位は facts にする

- `カテゴリ`
- `productArchetype` に相当する作りたいものの型
- `targetSummary`
- `teamSize`
- `timeline`
- `integration`
- `usageSurface`
- `audienceScope`
- `dataSensitivity`

### 3. 問い合わせは「最後の送信口」に寄せる

- 軽い相談口に見せない
- 原則として、問い合わせの前に詳細見積または同等の自動整理を通す
- 問い合わせページ直流入でも、内部的には同じ整理基準へ寄せる
- 共通 language は [`チャット共通facts一覧.md`](./チャット共通facts一覧.md)、導線ごとの役割は [`チャット質問棚卸し表.md`](./チャット質問棚卸し表.md) を参照する

## Phase 1: 入口の再設計

### 目的

- 費用起点の導線を、より自然な「欲しいもの」起点へ寄せる

### 変更方針

- トップ・主要チャットの最初の問いを、次のようなカテゴリへ再設計する
  - お客様との接点を作りたい
  - 顧客や案件の管理を楽にしたい
  - 社内作業を減らしたい
  - 予約や受付を自動化したい
  - 独自のサービスを形にしたい
  - まだうまく言えないので整理したい

### 補足

- 「費用を知りたい」「依頼方法を知りたい」は入口ではなく、途中の意図確認や CTA で扱う
- 入口の役割は、温度感の判定ではなく**案件カテゴリの正規化**に寄せる

### 主な対象

- `docs/top-chat-branching-design.md`
- `docs/分岐別解答マップ.md`
- `src/lib/chat/concierge-flow.ts`

---

## Phase 2: カテゴリ別の深掘り質問を再設計

### 目的

- 質問数を増やさずに、見積・要件整理に効く情報密度を上げる

### 変更方針

- 2 問目は「どこまでできれば十分か / 最高か」を身近な言葉で聞く
- 3 問目以降で、見積に効く制約だけを聞く

### 例

- Web / 接点系
  - 名刺代わり
  - 営業マン
  - 店舗窓口
- 顧客管理系
  - メモできれば十分
  - 集計や予約まで自動化したい
  - 通知や予測までやりたい

### 質問設計ルール

- 専門用語を UI に出さない
- 選択肢ごとに内部の技術用語・要件用語へ翻訳する
- 自由記述は補足として扱い、数値や送信可否の主判定に使いすぎない

### 主な対象

- `src/lib/content/site-copy.ts`
- `docs/estimate-wizard-core-axes-required-flow.md`
- `src/lib/inquiry/inquiry-brief.ts`

## Phase 3: facts モデルの共通化

### 目的

- 導線ごとに見た目が違っても、再質問の基準を揃える

### やること

- 既存の `track/path` と `EstimateFormDraft` の間で、意味的に重なる項目を棚卸しする
- 入口別回答を、共通 facts へ変換するマッピング表を用意する
- `answeredQuestionIds` / `prefilledQuestionIds` だけでなく、**意味の重複**を避けるルールを明文化する
- 現状把握の起点として [`チャット共通facts一覧.md`](./チャット共通facts一覧.md) と [`チャット質問棚卸し表.md`](./チャット質問棚卸し表.md) を更新運用する

### 完了条件

- 「demo で答えた内容を問い合わせで別の言い方で再質問する」ケースが潰せる
- どの入口でも、不足分だけを聞くフローへ寄せられる

### 主な対象

- `src/lib/chat/estimate-handoff.ts`
- `src/lib/journey/visitor-journey-estimate-prefill.ts`
- `src/lib/contact/parse-estimate-answers-to-form-draft.ts`

## Phase 4: 問い合わせ送信ゲートの強化

### 目的

- 情報収集・比較目的の送信を減らし、初回返信の品質と効率を両立する

### 変更方針

- 問い合わせ送信前に、以下を必須に寄せる
  - 仮要件定義の確認
  - 概算レンジの確認
  - 意図確認の最終 1〜3 問

### 最終確認で入れたい質問候補

- 今回、まず何を判断したいですか
- この内容を社内で誰が判断しますか
- 概算を見たうえで、次に相談したいことは何ですか

### 送信を止める基準

- 情報収集が主目的
- 相見積もり比較だけが目的
- 案件の対象業務が不明
- 整理結果をまだ確認していない

### 主な対象

- `src/lib/contact/contact-intake-satisfaction.ts`
- `src/components/contact/ContactForm.tsx`
- `src/lib/inquiry/inquiry-brief.ts`
- `src/lib/content/site-copy.ts`

### 実装メモ（反映済み）

- 共通ゲート判定を `src/lib/inquiry/inquiry-brief.ts` の `evaluateInquiryGate` に集約
- 見積後 CTA (`src/components/estimate/EstimateDetailedAmountContent.tsx`) は上記判定の `sendable` のみ有効
- 直 `/contact` は `src/components/contact/ContactForm.tsx` で `ContactIntakeHearingBlock` を配線し、ヒアリング完了後に synthetic snapshot + inquiry brief を作成
- 送信ボタンは共通判定に連動し、未完了時は送信不可のまま補足案内を表示

## Phase 5: demo へ流す判断基準の反映

### 目的

- 問い合わせに送らない判断を、ネガティブではなく自然な価値導線として見せる

### やること

- 情報収集段階の人には `demo/list` / `experience` / `services` を優先する条件を定義する
- チャット完了時の CTA 優先順位を見直す
- 「相談前に見たほうが早い」ケースでは、問い合わせではなく demo へ送る

### 代表ケース

- 画面イメージを先に見た方が早い
- 要件の言語化より、近い成果物の確認が先
- まだ技術や事例を見たい段階

### 主な対象

- `docs/top-chat-branching-design.md`
- `docs/分岐別解答マップ.md`
- `src/lib/chat/concierge-routing.ts`
- `src/lib/chat/concierge-flow.ts`

## Phase 6: 文言とブランドトーンの修正

### 目的

- 「軽い無料相談窓口」に見えるコピーを、整理済み案件向けの窓口へ寄せる

### 修正方針

- `無料相談` の露出を減らす
- `整理済みの相談`, `前提確認`, `概算`, `社内共有用のたたき台` を前面に出す
- 「まず話だけでも」を避ける
- 「わからない項目は大丈夫」だが「何も決めなくてよい」とは言わない

### 主な対象

- `src/lib/content/site-copy.ts`
- `docs/要件定義.md`
- `docs/chat-reply-templates.md`

---

## Phase 7: 初回返信運用のテンプレ化

### 目的

- 問い合わせ体験だけでなく、受信後の運用まで一体で軽くする

### やること

- 初回返信テンプレを、以下の構成で固定する
  - 受領確認
  - 自動整理結果の前提再掲
  - 概算レンジまたは前提条件の再掲
  - 必要なら 1〜3 個の確認事項
  - 進める場合のみ 30 分相談案内

### 完了条件

- 本気度の低い相手にはテンプレで十分
- 本気度の高い相手だけが 2 通目以降で具体化する

### 主な対象

- `docs/chat-reply-templates.md`
- メール送信まわりのテンプレ管理箇所

## 優先順位

### 先にやる

1. 問い合わせポリシーの固定
2. 問い合わせ送信ゲートの強化
3. demo / 問い合わせ振り分け基準の明文化
4. 入口カテゴリの再設計

### 後でもよい

1. 細かな文言の研磨
2. 返信テンプレの微調整
3. 係数や相場テーブルの精緻化

---

## 受け入れ条件

- 問い合わせページは、**概算確認前の軽い相談窓口**に見えない
- demo / 体験へ流す条件が、ドキュメントとチャット設計で一致している
- 直問い合わせでも、最低限の案件具体化なしでは送信できない
- トップ / demo / services から来ても、同趣旨の再質問が減る
- 初回返信をテンプレ中心で返せる情報量が、送信時点で揃う

---

## 実装メモ

- この PLAN は、いきなり全面実装するより **Phase 4 → Phase 5 → Phase 1/2** の順でも進められる
- 先に送信ゲートと問い合わせコピーを締めるだけでも、運用負荷はかなり下がる
- その後に入口カテゴリと facts モデルを揃えると、全チャットボットへの反映がしやすい
