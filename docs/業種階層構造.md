# トップ／コンシェルジュ用 業種の階層化・拡張 PLAN

## 1. 目的

- チャット選択肢の**業種カバレッジ**を上げる（特に **人材・外食・食品卸** をマストで扱えるようにする）。
- 一方でトップ導線の**認知負荷とタップ数**を抑え、フラットにボタンを増やしすぎない。
- デモ一覧の絞り込み・Sanity の業種表・コンシェルジュのスコアリングが**同じ語彙で整合**するようにする。

## 2. 現状（参照）

- **実装契約（handoff・answers キー）**: [industry-estimate-handoff-contract.md](industry-estimate-handoff-contract.md)
- コンシェルジュの事業領域: `src/lib/demo/intelligent-concierge.ts` の `CONCIERGE_DOMAIN_OPTIONS`。
- Sanity AI Demo の `industry`: `src/sanity/schemaTypes/aiDemo.ts`（一覧が限定的な場合、CMS とコンシェルジュの ID がズレやすい）。
- 見積ヒアリングの「業種」は `src/lib/estimate/estimate-detailed-answer-order.ts` 先頭キーなど別経路。

## 3. 方針（推奨）

### 3.1 2 層構造

1. **第 1 層（必須・少数）**  
   いまの「大分類」に近いが、**マスト業種が「その他」に吸い込まれない**よう、次のどちらかを採用する。
   - **A**: 大分類を増やす（例: 「人材・派遣」「飲食・外食」「食品・卸・流通」を第 1 層に明示）。
   - **B**: 大分類は据え置きつつ、**「サービス・小売・医療・飲食など」を分割**し、飲食・人材・卸を第 1 層に独立させる。

2. **第 2 層（任意・折りたたみ）**  
   「詳しく選ぶ（任意）」で細目を表示。**スキップ可**。スキップ時は第 1 層 ID のみでスコアリング継続。

3. **自由記述（任意・並列）**  
   第 1 層と**同じ画面内**に置き、**1 行入力を優先**（フォーカス時や必要時のみ複数行に拡張してよい）。プレースホルダで「清掃・ビルメン」「派遣・登録支援」など具体例を示す。

### 3.1.1 スマホ UI 方針（確定）

- **第 2 層は初期状態で閉じる**（アコーディオン等）。開いたときだけ縦スクロールが増えることを許容。
- **自由記述は 1 行優先**で、第 1 層＋（閉じた第 2 層）＋主 CTA が **だいたい 1〜1.5 画面**に収まりやすくする。
- **主 CTA（次へ／確定）は画面下部に固定（sticky）**し、軽いスクロールでも操作を迷わせない。

### 3.2 変更のおき場所（コード対応）

- **トップページのチャット**: [`src/components/chat/HomeConciergeFlow.tsx`](../src/components/chat/HomeConciergeFlow.tsx)  
  - いまは **`root`（知りたいこと）** の次がトラック別の Step 2（例: トラック A なら `A_STEP_BUILD`）。  
  - **業種 UI をトップで取る場合**の自然な位置は、**`root` 確定直後・トラック別質問より前の共通ステップ**（＝ユーザーさんの言う「目的を選んだあとの最初の画面」に相当）。
- **デモ一覧のコンシェルジュ**: [`src/components/chat/DemoListConciergeFlow.tsx`](../src/components/chat/DemoListConciergeFlow.tsx)  
  - 現状、**step 0 が事業領域**（`CONCIERGE_DOMAIN_OPTIONS`）。
- **定義の単一化**: 上記どちらに置くにせよ、**第 1 / 第 2 層のマスタと ID** は `intelligent-concierge` 側（または共通モジュール）に**1 箇所**に寄せ、トップ／一覧で二重定義しない。

### 3.3 ID 設計

- `ConciergeDomainId`（または同等）を **第 1 層と第 2 層で名前空間を分ける**（例: `domain` + 任意の `domainDetail`）。
- 既存 ID の意味を変えない（後方互換）。新規は追加のみ、廃止は別リリースでマイグレーション表を用意。

### 3.4 デモ・CMS との同期ルール

- コンシェルジュで選んだ値 → デモ一覧フィルタに渡せる **正規化マップ**（1 ファイルに集約推奨）。
- Sanity の `industry` オプションを増やす場合は、**コンシェルジュ ID との対応表**を `docs` またはコードコメントで固定する。

## 5. 非目標（スコープ外にしやすいもの）

- 全業種を第 1 層にフラット列挙すること。
- 見積フォームの質問数そのものの削減（別ドキュメント `0401見積もりUIUX見直し.md` で扱う）。

## 6. 受け入れ条件（例）

- 人材・外食・食品卸が、**第 1 層または明確な第 2 層**から 2 タップ以内で選べる。
- 第 2 層をスキップしてもフローが完了し、デモ推薦が破綻しない。
- 既存ユーザーの「特定せずに進む」動線が維持される。

## 7. 備考：詳細見積もりへの遷移時、業種まわりの重複質問をしないための手順

トップ（または一覧コンシェルジュ）で業種・補足文まで取ったあと **詳細見積もり**（[`EstimateDetailedHearingWizard`](../src/components/estimate/EstimateDetailedHearingWizard.tsx) 等）へ進んだとき、[`ESTIMATE_DETAILED_ANSWER_KEY_ORDER`](../src/lib/estimate/estimate-detailed-answer-order.ts) 先頭の **「業種」** と二重に聞かないようにする。実装時は次の順で進めると抜け漏れが減る。

1. **文脈に何を載せるか決める**  
   - 第 1 層 ID、第 2 層 ID（あれば）、自由記述 1 行を、見積ページが復元できる形で保持する。  
   - 既存の [`ConciergeEstimateContextPayload`](../src/lib/chat/estimate-handoff.ts)（`track` / `path` / `detailBlock`）に載せるか、**`path` に業種専用の `FlowSelection` を 1 件追加する**か、payload に任意フィールドを増やすかを選ぶ。  
   - 方針: **表示用サマリは `path` や `detailBlock` と一貫**させ、推論用は正規化 ID を別キーで持ってもよい。

2. **見積ページ入口でデコードする**  
   - [`buildEstimateDetailedEntryUrl`](../src/lib/chat/estimate-handoff.ts)／sessionStorage 経由で渡った payload を読み、**初期 `answers` レコード**（またはセッション初期状態）にマップする。

3. **ヒアリング側で「既に答えがあるキー」をスキップ or 読み取り専用**  
   - `answers["業種"]`（および第 2 層・自由記述をどのキーに載せるか）を事前入力済みにする。  
   - UI は **当該ステップを非表示**するか、**1 行で「コンシェルジュの選択: …」と表示して編集は任意**にする（後者は変更したいユーザー向け）。

4. **自由記述と見積の他キーの被りを決める**  
   - トップの補足が「いまいちばんやりたいこと・課題」と意味的に被る場合は、**マージ**（1 キーに連結）か **業種は業種キーにのみ**か、ルールを 1 本に固定する（`0401見積もりUIUX見直し.md` の重複排除方針と整合）。

5. **ドキュメント**  
   - [`docs/concierge-chat-scopes.md`](concierge-chat-scopes.md) に、見積 handoff で運ぶキーとセッション境界の変更があれば追記する。

**注意**: トップを経由せず詳細見積だけ開くユーザーは従来どおりフル質問となるため、**「handoff あり」のときだけ**スキップロジックを効かせる。
