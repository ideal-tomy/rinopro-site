# 要件定義・コンシェルジュ関連ドキュメントのリファクタリング計画

## 1. いま起きていること（問題の整理）

### 1.1 「要件としてそうなっている」のか「干渉」のか

**結論: ドキュメント上は「トップ専用の挙動」が十分に書かれておらず、実装側にだけ複数の特殊ルールが積み上がっている状態に近い。**

| レイヤ | 内容 |
|--------|------|
| `docs/要件定義.md` §3 | トップのチャットを「プリセット3つ・短い回答・2ラリーで誘導」と**古い・単純な仮定**で書いている |
| 実装 | `ChatContainer` が **pathname・`conciergeSurface`・`messages.length`・mode** の組み合わせで UI を切替（トップは広幅維持など**後追い修正**あり） |
| 実装 | トップの選択式は `HomeConciergeFlow`＋`concierge-flow.ts`（**分岐別解答マップ**・**top-chat-branching-design** が実体に近い） |
| 実装 | フリーチャットは `useChat`＋API。`default`＋`pageContext=top` では **シニアモードが原則発火しない**（`concierge-senior-mode.md` 明記） |
| 実装 | `ChatBubble` は **Markdown／リンク非レンダリング**（プレーンテキスト） |

→ **「他チャット定義が干渉」**というより、**一本の「プロダクト要件」がなく、`要件定義.md` と `分岐設計／スコープ／シニア` が別列車**で走っているのが、認識ズレと「直したはずが別バグ」の温床になっている。

### 1.2 トップだけ挙動が変に見える主因（実装の真実・要約）

以下は **バグというより設計の組み合わせ**として `要件定義` に明文化されていなかった点である。

1. **ウィザード → メッセージ1件以上で UI がチャット一覧専用に切り替わる**（`ChatContainer` の `mainContent` 分岐）。「戻る」「おすすめカード」はウィザード専用 UI に載っているものが多い。
2. **`/demo/list` のおすすめオーバーレイ**は `DemoListConciergeFlow` 専用で、**トップのフリーチャットには存在しない**。
3. **シニアモード**は `inferSeniorEngagement`＋`isSeniorEligibleContext` で **`top`＋`default` は対象外**（事例流入など例外あり）。→ トップ追質問は「薄いプロンプト＋フッター3リンク」になりやすい。
4. **アシスタント本文のリンク**は UI が解釈していないため、モデルが `` `/demo/list` `` と書いても **クリック不能**に見える。

これらを **要件に「意図として許容するか／直すか」**で一度決めないと、実装とドキュメントの両方が終わらない。

## 2. ゴール: 誤解が起きにくいドキュメント構造

### 2.1 「一つにまとめる」の意味の再定義

**単一ファイル1本に全章をマージすることは非推奨。**  
理由: `分岐別解答マップ` のような**長大なコンテンツ**と `sessionId` のような**実装契約**を同じファイルに混ぜると、更新が止まり、かえって「どれが正か」が分からなくなる。

代わりに次の **2層＋索引1本** を推奨する。

| 層 | 役割 | 読者 |
|----|------|------|
| **A. プロダクト要件（短い）** | 何をユーザーに約束するか、受け入れ条件（検証可能な一文） | PM・デザイン・実装の共通言語 |
| **B. 実装仕様（分割可）** | 分岐表・session 規則・シニア閾値・コード参照 | 実装・レビュー |
| **C. 索引（1ファイル）** | 「公式の読む順番」「変更したら同期するドキュメント一覧」 | 全員の入口 |

**「一つ」＝ 索引とプロダクト要件で「正の入口が一つ」**にする、と解釈するのが現実的である。

### 2.2 推奨ファイル配置（リファクタ後イメージ）

```
docs/
  要件定義.md                    … A の一部（サイト全体コンセプト＋チャットは要約のみ）
  concierge-spec-index.md        … C（索引・フェーズ1で整備済み）
  top-chat-branching-design.md   … B（トップ分岐の操作設計・既存）
  分岐別解答マップ.md             … B（文言・出力順・UI導線・コード参照・既存）
  concierge-chat-scopes.md       … B（sessionId 契約・既存）
  concierge-senior-mode.md       … B（シニア発火・既存）
  concierge-analytics-events.md  … B（任意）
```

**新規に追加するなら** `docs/concierge-product-requirements.md`（A のチャット部分を `要件定義` から移し、受け入れ条件を箇条書きで固める）を検討。  
ただし **フェーズ1では `要件定義.md` 内に要約＋索引へのリンクだけ**でもよい。

## 3. `要件定義.md` の直し方（提案）

### 3.1 残すもの

- §1 サイトコンセプト、§2 ページ構成、§4 遷移、§5 導線原則、§6 必須テキスト  
  → **文言は最小限の修正**（「完璧」など測れない表現は、必要なら「品質バーは別紙の受け入れ条件に従う」へ）

### 3.2 置き換えるもの（§3 チャットボット仕様）

現状の §3.2〜3.5（プリセット3つ・2ラリー等）は **実装と不一致**のため、次のどちらかにする。

**推奨（段階的）**

1. **§3 を「目的＋非機能の約束」に縮約**（5〜15 行）
   - 例: 営業トーン禁止、体験・確認優先、主導線はフッター CTA＋必要なら本文リンク（※リンクは UI でクリック可能であることを要件化するか明記）
2. **詳細は `concierge-spec-index.md` から辿る**と明記
3. **受け入れ条件（Must）を 3〜7 個だけ箇条書き**で §3 の末尾に追加（後述サンプル）

**受け入れ条件の例（ドラフト・要合意）**

- トップから開いたコンシェルジュで、**ウィザード完了後にフリーチャットへ移行した場合も**、モーダル幅・タイポグラフィが **他ページの同一コンポーネントと同等**である（狭い別 UI に見えない）。
- モデルがサイト内パスを案内する場合、**ユーザーがコピーせず辿れる**（Markdown リンクのレンダリング、または同等のクリック可能要素）。
- トップの選択式フローで提示する **概算・次の一手**は `分岐別解答マップ.md` と実装が一致する（変更は表とコードを同 PR で更新）。
- **シニア相当の応答厚み**をトップ default で求めるか否かを要件で明示し、`concierge-senior-mode.md` の発火条件と矛盾させない。

## 4. 既存ドキュメント間の関係（整理表）

| ドキュメント | いまの役割 | リファクタ後の位置づけ |
|--------------|------------|------------------------|
| `要件定義.md` | サイト全体＋古いチャット概要 | **短い約束＋索引へのポインタ** |
| `top-chat-branching-design.md` | トップ分岐の操作・CTA 設計 | **B の設計ソース**（変更時は分岐マップ・コードとセット） |
| `分岐別解答マップ.md` | 文言・出力順・結果 UI 導線 | **B のコンテンツソース**（`concierge-flow.ts` とペア） |
| `concierge-chat-scopes.md` | sessionId 規則 | **B の契約**（`concierge-session-id.ts` とペア） |
| `concierge-senior-mode.md` | シニア発火・プロンプト | **B の契約**（`concierge-senior.ts` とペア） |
| `chat-reply-templates.md` | テンプレ | 必要なら索引からリンクのみ |
| `concierge-analytics-events.md` | KPI | 索引からリンク |

**重複の扱い:** `top-chat-branching-design` と `分岐別解答マップ` は役割が違う（前者＝フロー設計、後者＝生成物・文言）ため **統合は不要**。索引で「どちらを先に読むか」を固定する。

## 6. 成功の定義

- 新規参加者が **「トップのチャットだけなぜ違うのか」**を、`要件定義` → `concierge-spec-index` → 該当 B 文書 の **30分以内**で説明できる。
- チャット関連の変更 PR で、レビュー観点が **「索引の同期表に書いた隣の doc を更新したか」**で機械的に確認できる。
- `要件定義.md` に **実装と矛盾する具体数値・画面仕様**が残っていない（詳細は B へ）。
