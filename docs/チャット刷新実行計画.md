# チャット刷新実行計画

## 目的

- [`チャット共通facts一覧.md`](./チャット共通facts一覧.md)、[`チャット質問棚卸し表.md`](./チャット質問棚卸し表.md)、[`問い合わせポリシー.md`](./問い合わせポリシー.md)、[`問い合わせ改善PLAN.md`](./問い合わせ改善PLAN.md) を土台に、**理想的な AI コンシェルジュ主導型 Web サイト**へ段階的に移行する。
- チャットの質問設計、自由記述、見積、問い合わせ、handoff、初回返信の流れを**同じストーリー**でつなぐ。
- 継ぎ足しで肥大化した箇所を整理し、**今後のアップデートがしやすい構造**へ寄せる。

---

## この計画の前提

### 1. `8文字以上` 制限は最終的に排除する

- 進行条件や送信条件を**文字数**ではなく、**必要 facts が揃っているか**で判断する構造へ移行する。
- `summary` の役割は、文字数で足切りするものではなく、`productArchetype` と `problemSummary` の整理に再定義する。

### 2. 選択式中心は維持する

- 基本 UI は「1画面1回答」「選択式中心」を維持する。
- ただし自由記述と音声入力は、例外ではなく**言語化支援の本丸**として扱う。
- 自由記述は canonical facts へ直接保存せず、まず `freeformMemo` として受け止め、AI / ルールで facts 候補へ変換する。

### 3. 全面書き直しはしない

- 既存の良い境界は残す。
- いきなり全導線を一斉変更せず、**フェーズごとに PLAN 作成 → 実行 → 回帰確認**で進める。

### 4. フェーズ単位で Plan mode を使う

- この実行計画書は「親計画」とする。
- 実作業は **1フェーズごとに**、以下の順で進める。

1. Plan mode で、そのフェーズ専用の PLAN を作成する
2. 依存関係・対象ファイル・受け入れ条件を明確にする
3. Agent mode で実装する
4. ドキュメント同期と回帰確認を行う
5. 次フェーズへ進む

---

## ゴール像

### ユーザー体験

- どのページから入っても、最初に「欲しいもの」や「困っていること」の言語化が進む
- 自由記述や音声入力でも、AI が内容整理を支援してくれる
- 同じ意味の質問を繰り返さず、不足分だけが聞かれる
- 詳細見積で仮要件定義と概算レンジが作られる
- 問い合わせ前には「この前提で相談する」という意思決定が伴う

### 内部構造

- 各チャット導線は別UIでも、裏側では共通 facts を使う
- `ChatContainer` などのシェル層は薄く、質問定義・CTA・handoff・facts 抽出は別境界に分かれている
- 質問項目変更や導線追加のたびに、複数箇所を手で同期しなくてよい

### 運用

- 初回返信はテンプレ中心で返せる
- 情報収集・比較目的の軽い問い合わせは demo / 見積で止まりやすい
- 本気度の高い相手だけが 2 通目や 30 分相談へ進む

---

## 進め方の原則

### 原則1: 先に構造、後で文言

- まず facts と責務境界を整える
- コピーや文言調整は、その上で行う

### 原則2: 先に受け皿、後で入口最適化

- 問い合わせ送信ゲート、詳細見積、handoff を先に安定させる
- その後にトップや demo の入口質問を再設計する

### 原則3: freeform は後回しにしない

- 自由記述は最後に足す機能ではない
- facts 抽出・確認質問・要約の流れを早い段階で整える

### 原則4: 1フェーズでやることを増やしすぎない

- 1フェーズで「構造変更」と「質問全面刷新」と「文言大改修」を同時にやらない
- 各フェーズの主目的を1つに絞る

- 実装前に毎回参照すべき計画が一つにまとまっている

## Phase 1: facts モデルの固定

### 目的

- 全チャットの裏側で扱う canonical facts を確定する
- 以後の質問変更や導線変更の単位を、文言ではなく facts にする

### このフェーズでやること

- `チャット共通facts一覧.md` の canonical facts をコード上の責務に落とす
- `productArchetype`、`problemSummary`、`targetSummary`、`inquiryIntent` などの中心 facts を確定する
- `visitorJourney`、`EstimateFormDraft`、`InquiryBrief` の役割境界を整理する
- facts の更新元と利用先を表にする

### このフェーズでまだやらないこと

- UI の全面変更
- トップ質問文の最終調整
- CTA 文言の最終調整

### 主な成果物

- facts のコード設計方針
- facts の更新経路一覧
- `8文字ルール` を置き換えるための最低 facts セット

### 完了条件

- 「何を聞くか」ではなく「どの fact を埋めるか」で設計会話ができる
- 各導線の回答が、どの facts を更新するか説明できる

---

## Phase 2: チャット shell と責務境界の整理

### 目的

- `ChatContainer` などの集約点を薄くし、今後の変更影響範囲を小さくする

### このフェーズでやること

- `ChatContainer` の責務を棚卸しする
- panel 分岐、CTA 判定、session 境界、導線レジストリの境界を引き直す
- 表示 shell とフロー定義と副作用を切り分ける

### 重点

- `ChatContainer`
- `concierge-panel-derived-state`
- `use-concierge-container-effects`
- CTA 判定ロジック

### このフェーズでまだやらないこと

- トップ質問の中身変更
- freeform の高度な AI 抽出

### 完了条件

- 新しい導線を足すとき、`ChatContainer` の条件分岐を大きく増やさずに済む
- CTA 仕様変更時に、複数ファイルへ散って追わなくて済む

---

## Phase 3: 質問定義の共通化

### 目的

- トップ、demo、services が別UIでも、質問の意味モデルを揃える

### このフェーズでやること

- 各導線の質問を `emitsFacts` ベースで棚卸しする
- `productCategory`、`productArchetype`、`desiredOutcomeLevel` の扱いを統一する
- トップ / demo / services の質問定義を registry 的に管理できる形へ寄せる

### 重点

- `チャット質問棚卸し表.md` の更新
- 質問定義ファイルの命名・責務統一
- optionId と handoff / routing / analytics の同期方針

### このフェーズでまだやらないこと

- すべての UI 文言の仕上げ
- 問い合わせ送信ゲートの最終実装

### 完了条件

- 「どの質問がどの facts を出すか」が一目で分かる
- トップと demo で、意味的に同じことを別言語で持たなくて済む

---

## Phase 4: freeform / 音声入力パイプラインの整備

詳細 PLAN: [`freeform音声入力パイプラインPLAN.md`](./freeform音声入力パイプラインPLAN.md)

### 目的

- 自由記述を正式な入力経路として扱えるようにする

### このフェーズでやること

- freeform を `freeformMemo` として保存する基準を統一する
- AI / ルールで facts 候補を抽出する流れを定義する
- 確信度の高いものだけ prefill し、曖昧なものは確認質問へ戻す
- 音声入力も同じ流れに載せる前提を作る

### 重点

- visitor journey への保存
- facts 抽出の責務
- 確認質問への戻し方

### 注意

- いきなり全自由記述を重い AI 判定にしない
- コストと遅延を見て、軽量ルール → 必要時のみ AI の順を基本とする

### 完了条件

- 自由記述が来ても、ただのメモで終わらず次の質問に活かせる
- 音声入力も同じ構造で扱える見通しが立つ

---

## Phase 5: 詳細見積の再設計

### 目的

- 仮要件定義と概算レンジの受け皿を、今後の中心基盤として強くする

### このフェーズでやること

- `8文字以上` 依存を撤去する
- `summary` を `productArchetype` と `problemSummary` に分離する
- 進行条件を文字数から facts 充足へ移行する
- 問い合わせへ渡す前提が揃うよう structured 質問を整理する

### 重点

- `EstimateDetailedHearingWizard`
- `EstimateFormDraft`
- `question-model`
- 関連コピー

### 完了条件

- 文字数制限なしで、必要情報が揃えば先に進める
- 自動要件定義・見積に必要な facts が structured に揃う

---

## Phase 6: 問い合わせ送信ゲートの再設計

### 目的

- 問い合わせを「軽い相談口」ではなく「整理済み案件の送信口」にする

### このフェーズでやること

- `問い合わせポリシー.md` に沿って送信条件を再設計する
- 概算確認後のみ送信しやすい流れへ寄せる
- 最後の本気度確認質問を設計する
- 問い合わせ直流入でも同じ基準へ寄せる

### 重点

- `inquiryBrief`
- `ContactForm`
- `contact-intake-satisfaction`
- 見積後の問い合わせ前整理

### 完了条件

- 情報収集・比較目的の送信が減りやすい
- 初回返信をテンプレ中心で返せる状態まで情報が揃う

---

## Phase 7: 入口質問の再設計

### 目的

- トップや各ページのチャットが、同じ最終受け皿へ自然につながるようにする

### このフェーズでやること

- トップの入口を「知りたい情報」起点から「欲しいもの」起点へ寄せる
- demo / services / global freeform の役割を再調整する
- 各導線の「次に demo / 見積 / 問い合わせのどれへ送るか」の基準を統一する

### 重点

- トップ分岐
- demo 一覧コンシェルジュ
- サービスカード導線
- freeform の受け皿

### 完了条件

- どのページから入っても、ストーリーが自然につながる
- 「この導線では何を整理する場所か」がユーザーにも内部にも明確

---

## Phase 8: 文言・返信テンプレ・運用調整

### 目的

- 構造変更の上に、ブランドトーンと運用効率を最適化する

### このフェーズでやること

- `無料相談` の露出や軽さを再調整する
- 問い合わせ・見積・チャット内 CTA の文言を統一する
- 初回返信テンプレを最終化する
- 運用観点で不要な質問・未使用文言を削る

### 完了条件

- UI の言葉と内部ポリシーが矛盾しない
- 運用側の返信コストが想定どおり軽い

## 各フェーズで必ず作る PLAN の型

各フェーズに着手する前に、Plan mode では最低限次を埋める。

1. そのフェーズの目的
2. 今回は何を変え、何を変えないか
3. 依存する前フェーズ
4. 触るファイル一覧
5. 受け入れ条件
6. 回帰確認項目
7. ドキュメント同期先

---

## 推奨実行順

### 最短で価値が出る順

1. Phase 1 `facts モデルの固定`
2. Phase 2 `chat shell と責務境界の整理`
3. Phase 5 `詳細見積の再設計`
4. Phase 6 `問い合わせ送信ゲートの再設計`
5. Phase 3 `質問定義の共通化`
6. Phase 4 `freeform / 音声入力パイプラインの整備`
7. Phase 7 `入口質問の再設計`
8. Phase 8 `文言・返信テンプレ調整`

### 理由

- 受け皿と境界を先に安定させた方が、後から入口を変えても壊れにくい
- `8文字ルール` 撤去は、詳細見積の再設計とセットで扱う方が安全
- freeform 強化は、facts モデルと確認質問の流れが固まってからの方が設計しやすい

---

## 成功条件

- `8文字以上` に依存しない
- 選択式中心でも、自由記述・音声入力が価値ある入力として機能する
- どのチャットから入っても、最終的な要件定義・見積・問い合わせへ自然につながる
- 変更時に「この文言を変えたらどこまで影響するか」が見えやすい
- 今後のアップデートが、フェーズ単位・責務単位で進めやすい
