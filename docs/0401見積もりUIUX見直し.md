# 0401 詳細見積もり UI/UX 見直し PLAN（質問効率 × 精度）

## 1. 目的

- **質問・入力の回数を減らす**（同じ内容の重複をなくす、まとめて聞く）。
- **回答しづらい項目はスキップ可能**にしつつ、概算の**信頼感**を落としすぎない。
- 既存のトーン（`src/lib/estimate/estimate-output-philosophy.ts` の「わからないは無理に決めなくてよい」）と矛盾しない。

## 1.1 必須要件（重複質問の禁止）

- **ユーザーがすでに答えた内容と同じ趣旨の質問を、見積ヒアリングで再度聞かない**（同一セッション内のウィザードだけでなく、**トップ／一覧コンシェルジュから詳細見積へ来た場合**に handoff で渡った項目も含む）。
- 実装レベルでは、**既に値が入っている `answers` キーに対応するステップは出さない／読み取り専用 1 行にする／省略**のいずれかをルール化する（詳細は `docs/業種階層構造.md` 第 7 節と整合）。
- 文言が違うだけの**二重ヒアリング**（例: 業種と「どんな事業か」の完全重複）は、キー設計またはマージルールで防ぐ。

## 2. 現状の整理（参照）

- 質問キーと表示順: `src/lib/estimate/estimate-detailed-answer-order.ts` の `ESTIMATE_DETAILED_ANSWER_KEY_ORDER`。
- 回答レコード生成: `src/lib/estimate/build-estimate-detailed-answers.ts`。
- UI: `EstimateDetailedHearingWizard.tsx`、モバイルシェル・結果表示まわり。
- 哲学文: `ESTIMATE_PHILOSOPHY_*`（不確実性の言語化は既に UI/Markdown 側にあり活用可能）。

## 3. 基本戦略

### 3.1 重複排除

- ラベルが違うだけで**同じ情報を二重に聞いていないか**を、`ESTIMATE_DETAILED_ANSWER_KEY_ORDER` を基に棚卸し。
- 統合する場合は **1 キーにマージ**し、結果ページ・スナップショット・API ペイロードのキー整合を一度に直す。
- **他導線からの入力**（コンシェルジュ完了文脈・URL／sessionStorage の handoff 等）と見積ウィザードの質問が**意味的に重なる場合は、ウィザード側を抑える**（§1.1 の必須要件）。

### 3.2 条件分岐（必要な人だけ深掘り）

- 例: 個人情報「含まない」なら、関連する深掘りを出さない。
- 例: 予算が未入力でもレンジ提示するなら、「予算の補足」と「ご予算のイメージ」の役割を明確に分けるか統合する。

### 3.3 1 画面・1 問での多項目入力

- 関連項目を**1 ステップに束ねる**（箇条書き・チェックリスト・短文 1 つで複数次元をカバー）。
- バリデーションは「全部必須」ではなく、**概算に効く最小セット**だけ必須にする。

### 3.4 スキップと精度の両立（出力契約）

- 未回答・「わからない」は **欠損**として扱い、結果側で:
  - **仮定**を短く明示する（「未入力のため ○○ と仮定」）。
  - **レンジを広げる**／**注意書きを付ける**など、ルールを `estimate-output-philosophy` または生成ロジック側で固定。
- ユーザーに見える形で **「確からしさ」や「未確定フラグ」**を統一表現する（結果 Markdown / `EstimateDetailedResultBody` 系）。

## 6. 成功指標（例）

- 中央値で**完了までのステップ数・入力フィールド数**が減る。
- 「わからない」選択後も、結果に**矛盾や無説明の単一点見積**が出ない。
- サンプル 1〜3 の期待文面と実出力がレビューで一致（回帰チェック用に表を残す）。
- **コンシェルジュ等で既に答えた項目**について、見積ウィザードで**同趣旨の質問が出ない**（回帰: handoff あり／なしの両方）。

## 7. 非ログイン時の「解答・行動の保持」と「最初から始めたい」の両立（検討メモ）

ログインがない前提では、**ブラウザを閉じる（タブを閉じる）まで**の範囲で、見積・コンシェルジュの途中経過を **`sessionStorage` 等で保持**しておくのは、重複質問の回避や再開体験のために**有力**（既存の `estimate-detailed` まわりのストレージ方針と揃える）。

一方で、保持が強すぎると **「新しくチャット／見積を始めたいのに、前回の途中からしか進めない」** と感じるユーザーが出る。次のような**明示的な逃がし**をセットで用意するのがよい。

- **「最初からやり直す」**（このサイト内に保存した当該フローの状態をクリアし、同一導線を step 0 から）。既存のイントロ／フロー初期化パターン（例: `clearEstimateDetailedFlow`）と役割を整理する。
- **入口の分離**: 「続きから」と「新規」が同じ URL だと誤解が生じやすいので、**トップから開き直したときは新規優先**にするか、再開はクエリや専用CTAに寄せるかを決める。
- **スコープの明示**: 「この端末のこのタブでのみ保持」と短く示し、**別タブ／シークレットでは共有されない**ことを期待値としてそろえる。
- **スキーマバージョン**: フォーム構造を変えたときは **storage キーまたはバージョン**を上げ、古い途中状態で**無言に途中ステップから始まる**ことを防ぐ。

最終方針は、プロダクトの「再開をどこまで尊ぶか」とトレードオフになるため、**§5 のチェックに「やり直し動線の手動確認」を必ず含める**。
