# チャット開始率を上げるUIUX提案

## 目的
- 「難しそう」という初期抵抗を下げ、チャット開始までの心理的距離を短くする。
- 説明ではなく操作へ自然に遷移させる。

## 課題仮説
- CTAを押す前に、ユーザーが「何を入力すればよいか」を決めきれない。
- AIの存在が待機状態で見えず、会話体験の温度が伝わらない。

## 実装メモ（現行コードの置き場所）
- 先回り吹き出しは、**全ページで共通化**し、**右下固定**の `ChatContainerLazy` 内（[`src/components/chat/ChatContainerLazy.tsx`](../src/components/chat/ChatContainerLazy.tsx)）で、FAB「**AIに相談**」**直上**に [`ConciergeFabNudge`](../src/components/chat/ConciergeFabNudge.tsx) として表示する。スクロール位置に依らず常に**同じ導線**に寄せ、発見性と「待機している感」を出す。
- コピーは [`src/lib/content/site-copy.ts`](../src/lib/content/site-copy.ts) の `conciergeFabNudgeShared` / `conciergeFabNudgeByPageId` / `getConciergeFabNudgePageId(pathname)` で **URL に応じて切替**。`localStorage` キーは `axeon:fab-nudge:v1`、状態は **ページID単位**（他ページの閉じた状態と独立）。
- 計測イベント名: `docs/concierge-analytics-events.md` の `fab_nudge_*`（`nudgePageId` を payload に含む）。
- **新規ルート**で専用コピーが要る場合の手順: 型 `ConciergeFabNudgePageId` を拡張 → `getConciergeFabNudgePageId` に分岐を追加 → `conciergeFabNudgeByPageId` に定義を追加。
- **手動確認の目安**: `/`・`/demo`・`/demo/list`・任意の `/demo/[slug]`・`/services`・`/contact` で吹き出しコピーが切り替わること（`npm run build` は通過済みの想定で、UIは dev で確認）。

## 提案1: AIの先回り問いかけを常時表示
- **右下「AIに相談」FAB** 付近（直上）に小さな吹き出しを常設する。
- 表示例: `こんにちは。いま一番面倒なことは何ですか？`
- 効果: 押下前に「会話が始まっている感覚」を作れる。

## 提案2: 入力例の常設で迷いを減らす
- プレースホルダではなく、CTA近接の固定テキストで例を見せる。
- 表示例: `例）写真整理に毎日20分かかる`
- 効果: 初回入力までの思考負荷を下げる。

## 提案3: 失敗不安を消す安心文言
- 表示例: `うまく言語化できなくても、質問に答えるだけで整理できます。`
- 効果: 入力への心理的安全性を確保する。

## 提案4: 「最初の1分で起きること」を可視化
- 3ステップを短文で提示する。
  - 悩みを1行で送る
  - AIが2-3問で整理する
  - 明日やることが出る
- 効果: 結果の予感を与え、開始判断を早める。

## 提案5: 表示の主導権をユーザーに渡す（非表示/停止/最小化）
- 吹き出しに `閉じる` `最小化` `切り替え停止` の操作を用意する。
- 一度閉じた状態は保存し、次回訪問時も尊重する。
- 効果: 再訪ユーザーの「邪魔」感を抑え、押し付け感をなくす。

## 提案6: モバイル専用の軽量表示ルール
- モバイルでは吹き出し文言を1-2行に制限する。
- ローテーション文言は最大2つまでに絞る。
- 一定時間後は自動でコンパクト表示に移行する。
- 効果: 画面占有率を抑え、可読性と操作性を両立する。

## 提案7: 再訪ユーザー向けの頻度制御
- 初回訪問: 通常表示（導線を明確化）
- 2-3回目: 小さめ表示（存在を示す）
- 4回目以降: 既定で最小化（必要時のみ展開）
- 効果: 学習済みユーザーへの過剰訴求を防ぎ、離脱要因を減らす。

## 提案8: 会話以外の探索導線を同時に提示
- 吹き出し内に `自分で見る` などの導線を置く。
- 例: デモ一覧、サービス説明、問い合わせページへのショートカット。
- 効果: 「話したくない」ユーザーの行動を止めずに回遊へつなげる。

## 提案9: 全ページ共通UI + ページ別文言最適化
- 同じ吹き出しUIを**上記のグローバル右下導線**でトップ、demo、demo一覧、サービス、問い合わせ等で共通利用する（**各ページに個別に埋め込む**のではなく、**1コンポーネント＋URL分岐**）。
- 表示文言のみページ文脈に合わせて切り替える（`nudgePageId` ベース。実体は `site-copy` の `getConciergeFabNudgePageId`）。
- 効果: UIの一貫性を保ちながら、各ページでの適合率を上げる。

### ページ別文言の例
- トップ: `いま一番面倒なことを1行で教えてください。`
- demo一覧: `目的に近いデモを選べば、会話の精度が上がります。`
- demo詳細: `このデモで検証したい点を一緒に絞りましょう。`
- サービス: `導入前提でなくてもOK。まずは悩み整理から始められます。`
- 問い合わせ: `うまく説明できなくても大丈夫。順番に質問します。`

## 表示優先度（実装順）
- 1位: AI吹き出し
- 2位: 入力例
- 3位: 安心文言
- 4位: 1分ステップ
- 5位: 表示制御（非表示/停止/最小化）
- 6位: モバイル軽量化
- 7位: 再訪頻度制御
- 8位: 会話以外の探索導線
- 9位: ページ別文言最適化

## A/Bテスト案（最小）
- A: 現状
- B: AI吹き出し + 入力例 + 安心文言
- 比較指標: CTAクリック率、チャット開始率、初回入力完了率

## A/Bテスト案（拡張）
- A: 常時表示のみ
- B: 表示制御あり（閉じる/最小化/停止） + モバイル軽量化
- C: B + ページ別文言最適化
- 比較指標: チャット開始率、初回入力完了率、再訪時の離脱率、ページ回遊率

## Cursor への PLAN 作成依頼文（トップ＋チャット開始率UI）

### このブロックの使い方
- **1回目の依頼**: 次の枠内テキストをまるごと Cursor に貼る。**PLAN だけ** を出させ、承認してから実装に進む。
- **2回目以降**（仕上げ・別ページ展開）: 同じ枠に「今回のスコープ（例: まずトップのみ / 次に demo 一覧」）を1行足して貼る。

### 枠内テキスト（コピー用）

次の条件で **実装用 PLAN** を作って。コードはまだ書かない（まず PLAN のみ）。

**前提・参照ドキュメント**
- `docs/トップ説明設計提案.md` … トップの意味付け（何者 / 価値 / 最初の行動 / 安心）と、表示の主導権・モバイル・再訪・回遊導線
- `docs/チャット開始率を上げるUIUX提案.md` … 本ファイル。吹き出し・入力例・1分ステップ・表示制御・ページ横断
- プロジェクトルール: `.cursor/rules/project-rules.mdc`（とくに UI/UX 前提・チャット会話スコープを変えるなら `docs/concierge-chat-scopes.md` も確認）
- 計測の足がかり: `docs/改善効果の計測設計提案.md`（イベント名の整合）

**ゴール**
- 初見がトップを開いた3秒程度で、サイトの性質と「最初にやるとよいこと」が分かる。
- チャット CTA まで到達しやすく、**押し付けがちなUI** にならない（閉じる/最小化/再訪配慮）。
- モバイルで吹き出しが画面を圧迫しない（簡易ルールに沿う実装方針を PLAN に含める）。

**今回のスコープ案（未指定なら次をデフォルト）**
- 現行実装では **FAB 直上＋全ページ分岐** は既に入っている。改修する場合の例: 文言 A/B、ローテ速度、`line-clamp`、z-index、未カバーURL の `nudgePageId` 追加。
- 大きな変更を**別PR**に分けるなら: (1) `site-copy` の文案のみ (2) `ConciergeFabNudge` の挙動・a11y。LLM の `role` 配線は `docs/チャット役割プロンプト設計提案.md` の**別依頼**とする。

**PLAN に必ず含める章**
1. 現状調査: トップ（および今回含めるなら他ページ）のルート・主要コンポーネント
2. UI 差分: 追加する要素の一覧（吹き出し、文言、CTA 周辺、ステップ可視化）
3. 状態管理: 閉じる/最小化/ローテ停止/再訪の頻度制御（localStorage キー案、命名、`prefers-reduced-motion` 時の扱い）
4. レスポンシブ: PC/モバイルでの文言行数・余白
5. a11y: 吹き出しの role/aria、キーボード、フォーカス
6. 受け入れ条件: 本ドキュメントの「実装チェックリスト」と整合する箇条書き
7. 計測: 取りたいイベント名と発火箇所（`docs/改善効果の計測設計提案.md` と揃えるかどうか）
8. リスクとロールバック: 一発で戻せる差分の切り方

**チャット中身（プロンプト・role 切替）について**
- 本タスクの PLAN は **UI とトップ導線が主**。**LLM の System/Developer 中身** は別タスク `docs/チャット役割プロンプト設計提案.md` と切り分ける。  
- 触る場合は「トップで開くチャットが `top_concierge` を使う**だけ**」の軽い配線に留め、専用ロールの本番実装は **別依頼**とする、と PLAN に明記する。

**出力形式**
- まず目次、次に上記1–8。最後に「推奨コミット分割（1 PR / 2 PR など）」。
