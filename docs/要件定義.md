# 要件定義

## 1. サイトコンセプト

### 1.1 基本方針

- **体験してください / 目で見て確認してください** をコンセプトに、UIUXで自然と誘導されるサイト
- テキストは最小限。営業っぽさを極限まで排除
- 技術力と実際に動くツール・見た人が使いたくなるdemoで「こんな活用方法があります」を紹介
- 技術力とコンサルタントとして実地を理解した開発が感じられる掲載内容
- UIUXを完璧に仕上げたサイト

### 1.2 事業背景

- 外資系コンサルから独立する28歳女性による事業
- DXツール・AIツールの開発と、企業内での有効活用のためのコンサルティング業務

### 1.3 ターゲット・ペルソナ

- 経営層・意思決定者（ITに疎い社長も含む）
- 建設業・士業など、現場業務の効率化・DX推進を検討している企業
- 「技術力」と「実地理解」の両立を求める層

## 2. ページ構成

| 順序 | ページ | 目的 |
|------|--------|------|
| 1 | トップ | 体験起点。チャットボットで初体験 |
| 2 | ツールdemo一覧 | 価値を体感。実際に触って確認 |
| 3 | 体験（/experience） | アプリに近い画面でイメージを掴く |
| 4 | サービス（開発 / コンサル） | 仕組みの説明 |
| 5 | 会社紹介 | 信頼補強 |
| 6 | 問い合わせ | 相談窓口 |

### 2.1 各ページの詳細

#### トップ
- **1行コピー**: ビジネスを再設計する
- **導線**: チャットボット体験 → ツールdemo一覧 / 体験ページ
- **狙い**: 初見で「何者か」を伝えつつ、体験で興味を喚起

#### ツールdemo一覧
- **1行コピー**: 実際に触って、導入後の業務変化を体感してください。
- **導線**: 各demoへの遷移 → サービス / 問い合わせ
- **狙い**: 使いたくなるツールを紹介し、活用イメージを具体化

#### 体験（/experience）
- **1行コピー**: まず触って、イメージを合わせてください。
- **導線**: 各体験の説明 → ツールdemo / サービス / 問い合わせ
- **狙い**: 導入実績の列挙ではなく、操作と体感で理解してもらう

#### サービス
- **開発**: 開発の流れの説明込み
- **コンサルティング**: 内容の説明と契約までの流れ説明込み
- **導線**: 問い合わせ
- **狙い**: プロセス透明性による安心感

#### 会社紹介
- **1行コピー**: 代表の戦略的知見と、パートナーの技術的資産。
- **導線**: 問い合わせ
- **狙い**: チームへの信頼構築

#### 問い合わせ
- **1行コピー**: あなたの現場の10分を、3秒に変える相談を。
- **狙い**: 売り込みではなく、**仮要件定義と概算確認を終えた案件の相談窓口**
- **補足**: 情報収集や比較だけの軽い相談は、原則として demo / 体験 / 詳細見積へ案内し、問い合わせは「整理済み案件の送信口」として扱う（詳細は [`問い合わせポリシー.md`](./問い合わせポリシー.md)）

## 3. チャットボット仕様

> **詳細仕様の正の入口**: [`concierge-spec-index.md`](./concierge-spec-index.md)（トップ分岐・解答マップ・会話スコープ・シニアモード・主要コードへの索引）。  
> **ドキュメント整理の経緯**: [`requirements-docs-refactor-plan.md`](./requirements-docs-refactor-plan.md)。

### 3.1 目的と非機能の約束

- トップ（およびサイト内の同一コンシェルジュ）で、**体験・確認**のトーンを優先する。営業導線にしない。
- 自然な会話体験を優先し、テキストは必要最小限に抑える。
- テキスト入力をデフォルトとし、音声入力はオフィス環境を考慮して任意でON可能とする。
- ウィザード型の選択式とフリーチャットは、**ページ・表面・会話段階**で UI が切り替わる。詳細は索引から辿る。

### 3.2 受け入れ条件（Must）

以下はプロダクトの検証可能な約束とする（変更時は本 § と索引・関連 doc を同じ更新単位で扱う）。

1. トップから開いたコンシェルジュで、**ウィザード完了後にフリーチャットへ移行した場合も**、モーダル幅・タイポグラフィが **他ページの同一コンポーネントと同等**である（狭い別 UI に見えない）。
2. モデルがサイト内パスを案内する場合、**ユーザーがコピーせず辿れる**（Markdown リンクのレンダリング、または同等のクリック可能要素）。
3. トップの選択式フローで提示する **概算・次の一手**は [`分岐別解答マップ.md`](./分岐別解答マップ.md) の記述と実装（`concierge-flow.ts`）が一致する（変更は表とコードを同 PR で更新）。
4. **シニア相当の応答厚み**をトップ `default` で求めるか否かを要件で明示し、[`concierge-senior-mode.md`](./concierge-senior-mode.md) の発火条件と矛盾させない。

### 3.3 実装詳細

分岐表・文言・session 規約・シニア閾値・コード参照は **[`concierge-spec-index.md`](./concierge-spec-index.md)** から辿る。旧 §3 にあった「2ラリー後の demo 誘導」等の方針は、**価値提供として demo/体験へ促す** を維持し、選択式とフリーチャットの差分の具体は B 層ドキュメントに委ねる。

## 4. ページ遷移のUX要件

### 4.1 参考: Speakライクの遷移

- 古いページと新しいページの **フェードアウト / フェードイン** を同期
- テキスト・ボタンの表示タイミングを遷移アニメーションと同期
- **ゆっくりページが移動してるのに遅く感じない** 技術を総結集

### 4.2 遷移の原則

| 要素 | 要件 |
|------|------|
| 古いページ | フェードアウト（適切なイージング） |
| 新しいページ | フェードイン（古いページの減衰と連動） |
| テキスト表示 | 遷移完了後、段階的に表示（遅延スタッガー） |
| ボタン表示 | テキスト表示と同期、またはその後に表示 |
| 体感速度 | 実際の遷移時間より短く感じさせる |

### 4.3 タイミング設計

- フェードアウト: 200〜300ms
- フェードイン: 300〜400ms（オーバーラップ可）
- テキスト/ボタン: 50〜100ms刻みのスタッガー
- 全体として「ぬるぬる動く」印象を維持

---

## 5. 導線設計の原則

- **主導線は常に1つ**: 1画面1アクション（迷わせない）
- **説明はインタラクションで代替**: ホバー、比較、ビフォーアフター
- **問い合わせは「売り込み」でなく「対話予約」のトーン**
- **問い合わせ前に整理を挟む**: 軽い相談窓口にせず、仮要件定義・概算・次の判断材料を揃えてから送信させる
- 各ページに「次に見る」導線を1つだけ明示

---

## 6. 必須テキスト（最小限）

テキストは削るが、以下は必須。

1. **1行タグライン**（トップに固定）
2. **各ページの1行目的文**
3. **次アクションの明示**（例: 次に見る / 相談する）
