# 要件定義

## 1. サイトコンセプト

### 1.1 基本方針

- **体験してください / 目で見て確認してください** をコンセプトに、UIUXで自然と誘導されるサイト
- テキストは最小限。営業っぽさを極限まで排除
- 技術力と実際に動くツール・見た人が使いたくなるdemoで「こんな活用方法があります」を紹介
- 技術力とコンサルタントとして実地を理解した開発が感じられる掲載内容
- UIUXを完璧に仕上げたサイト

### 1.2 事業背景

- 外資系コンサルから独立する28歳女性による事業
- DXツール・AIツールの開発と、企業内での有効活用のためのコンサルティング業務

### 1.3 ターゲット・ペルソナ

- 経営層・意思決定者（ITに疎い社長も含む）
- 建設業・士業など、現場業務の効率化・DX推進を検討している企業
- 「技術力」と「実地理解」の両立を求める層

## 2. ページ構成

| 順序 | ページ | 目的 |
|------|--------|------|
| 1 | トップ | 体験起点。チャットボットで初体験 |
| 2 | ツールdemo一覧 | 価値を体感。実際に触って確認 |
| 3 | 体験（/experience） | アプリに近い画面でイメージを掴く |
| 4 | サービス（開発 / コンサル） | 仕組みの説明 |
| 5 | 会社紹介 | 信頼補強 |
| 6 | 問い合わせ | 相談窓口 |

### 2.1 各ページの詳細

#### トップ
- **1行コピー**: ビジネスを再設計する
- **導線**: チャットボット体験 → ツールdemo一覧 / 体験ページ
- **狙い**: 初見で「何者か」を伝えつつ、体験で興味を喚起

#### ツールdemo一覧
- **1行コピー**: 実際に触って、導入後の業務変化を体感してください。
- **導線**: 各demoへの遷移 → サービス / 問い合わせ
- **狙い**: 使いたくなるツールを紹介し、活用イメージを具体化

#### 体験（/experience）
- **1行コピー**: まず触って、イメージを合わせてください。
- **導線**: 各体験の説明 → ツールdemo / サービス / 問い合わせ
- **狙い**: 導入実績の列挙ではなく、操作と体感で理解してもらう

#### サービス
- **開発**: 開発の流れの説明込み
- **コンサルティング**: 内容の説明と契約までの流れ説明込み
- **導線**: 問い合わせ
- **狙い**: プロセス透明性による安心感

#### 会社紹介
- **1行コピー**: 代表の戦略的知見と、パートナーの技術的資産。
- **導線**: 問い合わせ
- **狙い**: チームへの信頼構築

#### 問い合わせ
- **1行コピー**: あなたの現場の10分を、3秒に変える相談を。
- **狙い**: 売り込みではなく対話予約のトーン

## 3. チャットボット仕様

> **詳細仕様の正の入口**: [`concierge-spec-index.md`](./concierge-spec-index.md)（トップ分岐・解答マップ・会話スコープ・シニアモード・主要コードへの索引）。  
> **ドキュメント整理の段取り**: [`requirements-docs-refactor-plan.md`](./requirements-docs-refactor-plan.md)（§3 の現状と実装の差分整理・リファクタフェーズ）。

### 3.1 基本方針

- トップ画面に配置。体験のために使ってもらう
- 営業導線にしない。自然な会話体験を優先

### 3.2 回答ルール

- **回答は長文になりすぎない、的を得た解答**
- 1〜2回目の自由回答: 質問内容に応じた簡潔な回答のみ
- **2回以上会話をラリーした場合**: demo/事例への誘導を開始

### 3.3 誘導文言（2ラリー後）

営業感を出さず、価値提供として促す。

> ツールdemoや体験ページで近いものが見つかれば、あなたの希望は実現しやすくなります。まずは触ってイメージを合わせてみてください。

- 「体験してください」「目で見て確認してください」のコンセプトに沿った自然な促し
- 即座の他ページ誘導は行わない（2ラリー以上経過後にのみ）

### 3.4 入力方式

- テキスト入力: デフォルト
- 音声入力: 任意でON可能（オフィス環境を考慮）

### 3.5 初回表示

- 3つのプリセット質問を表示（自由入力はその下）
- 回答は短く、必要に応じてツールdemo・体験ページ（/experience）へのリンクを返す
- 資料請求・問い合わせの押し出しは行わない

## 4. ページ遷移のUX要件

### 4.1 参考: Speakライクの遷移

- 古いページと新しいページの **フェードアウト / フェードイン** を同期
- テキスト・ボタンの表示タイミングを遷移アニメーションと同期
- **ゆっくりページが移動してるのに遅く感じない** 技術を総結集

### 4.2 遷移の原則

| 要素 | 要件 |
|------|------|
| 古いページ | フェードアウト（適切なイージング） |
| 新しいページ | フェードイン（古いページの減衰と連動） |
| テキスト表示 | 遷移完了後、段階的に表示（遅延スタッガー） |
| ボタン表示 | テキスト表示と同期、またはその後に表示 |
| 体感速度 | 実際の遷移時間より短く感じさせる |

### 4.3 タイミング設計

- フェードアウト: 200〜300ms
- フェードイン: 300〜400ms（オーバーラップ可）
- テキスト/ボタン: 50〜100ms刻みのスタッガー
- 全体として「ぬるぬる動く」印象を維持

---

## 5. 導線設計の原則

- **主導線は常に1つ**: 1画面1アクション（迷わせない）
- **説明はインタラクションで代替**: ホバー、比較、ビフォーアフター
- **問い合わせは「売り込み」でなく「対話予約」のトーン**
- 各ページに「次に見る」導線を1つだけ明示

---

## 6. 必須テキスト（最小限）

テキストは削るが、以下は必須。

1. **1行タグライン**（トップに固定）
2. **各ページの1行目的文**
3. **次アクションの明示**（例: 次に見る / 相談する）
