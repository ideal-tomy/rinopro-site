# 公開後24時間監視 Runbook

対象: Vercel本番公開直後 24 時間  
目的: 障害の早期検知と、ロールバック判断の迅速化

## 1. 監視体制

- [ ] 監視責任者
- [ ] 技術一次対応
- [ ] 問い合わせ一次対応
- [ ] 連絡チャネル（Slack/電話）

## 2. 監視スケジュール

- 公開直後 0-2h: 15分ごと
- 2-8h: 30分ごと
- 8-24h: 1時間ごと

## 3. 監視項目（最重要）

- [ ] 5xx 発生件数（`/api/chat`, `/api/demo/run`, `/api/estimate-detailed`, `/api/inquiry-brief`, `/api/contact`）
- [ ] 429 発生件数（過剰制限/攻撃兆候）
- [ ] 問い合わせ到達件数（ログ運用の受付漏れ有無）
- [ ] 主要導線の表示可否（トップ、デモ、体験、問い合わせ）

## 4. 公開直後の再スモーク（必須）

- [ ] `/` 表示
- [ ] `/demo` 一覧表示
- [ ] `/demo/[slug]` 実行
- [ ] `/contact` 入力〜送信
- [ ] チャット導線で応答表示

## 5. 障害対応フロー

1. 検知（監視アラート/手動確認）
2. 影響範囲切り分け（全体/一部、API/画面）
3. 暫定対応（告知、代替導線案内）
4. 恒久対応可否を判断
5. ロールバック要否を決定

## 6. 即時ロールバック条件

- [ ] 主要導線停止が 15 分以上継続
- [ ] 問い合わせ不能が 10 分以上継続
- [ ] 5xx が連続し、復旧見込みが立たない
